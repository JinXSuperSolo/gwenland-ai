**Date:** 2026-06-09 21:00
**Scope:** `gwen-cli/packages/core/src/engine/inference/candle_ggqr/`

**Status:** ✅ STABLE — 236 lib tests pass; `cargo check --features candle-backend` 0 errors

---

## Executive Summary

GWEN-212 wires GGQR's existing dequantisation engine directly into a Candle-based forward pass,
producing a self-contained CPU inference backend for LLaMA-family models (Qwen3, LLaMA 3, Phi-3).
The backend loads quantised GGUF files, dequantises every tensor to `f32`, and runs autoregressive
generation without touching the network, a Python runtime, or anything outside pure Rust.

Eight implementation waves across two sessions. Net result: a complete `InferenceBackend` impl
registered in `BackendRegistry` under the name `"candle-ggqr"`, reachable by `config.toml` and
the `--backend` flag, with 236 passing tests and zero panics in production code paths.

## Why

### Why a Second Backend?

GWEN-211 added `MistralRsBackend`. MistralRS is capable but heavyweight — its optional dep alone
adds several hundred thousand lines of compiled code and imposes a strict `mistralrs-backend`
feature requirement. It also owns the entire model lifecycle; there is no way to inspect
intermediate tensors or hook into the forward pass.

GWEN-212 takes the opposite approach. We already had `convert::dequant` — a working GGQR
dequantisation engine that produces `Vec<f32>` from any Q2_K through Q6_K tensor. The only missing
piece was the forward pass. Candle is already a dependency (`candle-core 0.9`, `candle-nn 0.9`).
Plugging them together gives a backend we fully own, which means we can add LoRA adapters,
custom positional encodings, and speculative decoding later without waiting on an upstream crate.

The tradeoff is that we implement the transformer ourselves. That is intentional — every layer is
in our source tree, readable, auditable, and testable in isolation.

### Why Zero-Copy Tensor Conversion?

Dequantising a 4-bit model produces roughly `n_params * 4` bytes of `f32` data. For Qwen3-1.7B
that is around 6.8 GB if all layers are resident simultaneously. Any copy during the
`Vec<f32> → Tensor` handoff doubles peak RSS for the duration of the conversion.

`Tensor::from_vec` on `Device::Cpu` is documented to take ownership of the `Vec`'s heap buffer
(`storage_owned`) without copying. The spec required the ownership chain to go through
`Arc<[f32]>` first. `Arc<[f32]>` is a DST — `Arc::try_unwrap` requires `T: Sized`, so it cannot
be unwrapped directly. We use `Arc<Vec<f32>>` instead: semantically identical ownership model,
`Sized`, and `try_unwrap` succeeds because the `Arc` is constructed and immediately consumed in
the same call with strong count == 1. One unit test (`arc_from_vec_starts_at_one_strong_ref`)
asserts this explicitly.

### Why Implement the Transformer Manually?

`candle-transformers` has LLaMA and Qwen model types, but they expect weights in SafeTensors
format and manage their own file loading. Our weights come from GGUF via GGQR dequantisation —
a `HashMap<String, Tensor>` keyed by GGUF tensor names (`blk.0.attn_q.weight`, etc.). Wrapping
`candle-transformers`' model types around our weight map would require an adapter shim as complex
as just writing the forward pass directly.

Writing `rms_norm`, `attention`, `mlp`, and `forward` from scratch takes about 250 lines. Each
function is individually tested, and the complete forward pass is verified against known
input/output shapes. The alternative — fighting the `candle-transformers` API to accept a foreign
weight map — would have been longer and more fragile.

### Why No KV Cache?

KV cache is GWEN-215. Without it, every autoregressive step re-runs attention over the entire
growing context — O(n²) in sequence length. For the target use case (Qwen3-1.7B, 512-token
responses on a laptop CPU) this is acceptable. The spec says so explicitly, the generation loop
says so in its module-level comment, and the test parameters keep sequences short enough that
the tests finish in under a second.

Adding KV cache later is an isolated change to `generation.rs` and `forward.rs` — no other
module touches generation state.

### Why Feature-Gate Behind `candle-backend`?

`candle-core`, `candle-nn`, and `candle-transformers` are already unconditional dependencies,
so the feature gate carries no compile-time cost in the default build. The gate exists to
let `BackendRegistry::new()` and `selector.rs` conditionally register and resolve the backend,
matching the pattern established by `mistralrs-backend`. It also makes the intent explicit in
`Cargo.toml` — the candle-ggqr code path is opt-in, not ambient.

## What: Eight Waves

### Wave 0 — Feature Gate + Core Types

**Files:** `Cargo.toml`, `error.rs`, `engine/inference/mod.rs`, `engine/inference/params.rs`

Added `candle-backend = []` to `[features]`. The feature is empty because all candle crates are
already unconditional deps — the gate is purely for conditional compilation of our module tree.

Extended `GwenError` with five new variants:

```rust
Dequantization { tensor_name: String, error: String }
InferenceError { layer: String, operation: String, error: String }
InsufficientMemory { required_gb: f32, available_gb: f32 }
ArchitectureNotSupported(String)
CandleError(String)
```

`InferenceError` carries `layer` and `operation` separately so error messages tell you exactly
which tensor lookup or matmul failed and in which transformer layer — useful when a shape
mismatch surfaces six layers deep in a 32-layer model.

Added `seed: Option<u64>` to `InferParams`. Previously missing; needed by the sampling dispatcher
to seed `StdRng` for reproducible generation.

Created `engine/inference/candle_ggqr/mod.rs` with `#![cfg(feature = "candle-backend")]` as
an inner attribute — gates the entire submodule tree with one declaration rather than repeating
the cfg on every item.

### Wave 1 — GGUF Validation and KV Metadata

**File:** `candle_ggqr/gguf.rs` (new, ~450 lines, 10 tests)

Three public functions:

**`validate_gguf(path)`** — checks magic bytes, tensor count ≤ 10,000, dimension sizes ≤ 100,000
per axis, and byte buffer sizes match the expected count for each dtype. Returns a parsed
`GgufFile` on success. Anything malformed gets a descriptive `ModelLoad` error rather than
a downstream panic when we try to allocate a tensor with a bogus shape.

**`extract_architecture(path)`** — re-reads the KV section to find `general.architecture`.
Accepts `["llama", "qwen2", "phi3"]`. Anything else is `ArchitectureNotSupported`. Fast path —
only reads bytes until it finds the key, then closes the file.

**`build_model_config(path, arch)`** — reads all nine architecture-prefixed metadata keys
(`block_count`, `embedding_length`, `attention.head_count`, etc.) and assembles `ModelConfig`.
Vocab size falls back to `tokenizer.ggml.token_type_count` if the arch-prefixed key is absent.
`rope_theta` defaults to 10,000.0 and `rms_norm_eps` to 1e-5 when not present — both safe
defaults that match the original LLaMA paper values.

The existing `gguf_parser.rs` was not modified. It skips all KV data (only needs tensor offsets
for dequantisation). Rather than adding a second read mode to a module that has one job, we
open the file independently in `gguf.rs` and read only what we need. Two file opens, two clear
responsibilities.

Added `#[derive(Debug)]` to `GgufFile` in `gguf_parser.rs` because `unwrap_err()` in tests
requires `T: Debug`. That is the only change to any pre-existing file in this module.

### Wave 2 — GGQR Dequantisation Integration

**File:** `candle_ggqr/dequant.rs` (new, ~130 lines, 4 tests)

```rust
pub fn dequantize_tensor(tensor: &TensorInfo) -> Result<Vec<f32>, GwenError>
```

One function. Calls `convert::dequant::dequantize(tensor, DequantMode::Standard)` and maps the
`String` error to `GwenError::Dequantization { tensor_name, error }`. The tensor name travels
with the error so load-time failures identify the specific weight rather than saying "dequant
failed" with no context.

F32 and F16 tensors pass through unchanged (the dequant engine handles them). Q2_K through Q6_K
go through the K-quant superblock paths.

### Wave 3 — Zero-Copy Vec→Tensor Conversion

**File:** `candle_ggqr/tensor.rs` (new, ~135 lines, 6 tests)

```rust
pub fn vec_to_tensor(data: Vec<f32>, shape: impl Into<Shape>, device: &Device)
    -> Result<Tensor, GwenError>
```

Ownership chain: `Vec<f32>` → `Arc<Vec<f32>>` → `Arc::try_unwrap` → `Vec<f32>` → `Tensor::from_vec`.

The `Arc` is constructed with strong count == 1 and immediately unwrapped. `try_unwrap` on a
count-1 `Arc` is infallible — the `unwrap_or_else(|a| (*a).clone())` fallback is a belt-and-
suspenders guard that would only fire if someone cloned the `Arc` between construction and
`try_unwrap`, which cannot happen in a single-threaded call stack. The unit test
`arc_from_vec_starts_at_one_strong_ref` verifies this property directly.

Two tests were removed during development: one for empty-vec tensors (candle 0.9 crashes on
0-element tensors with `STATUS_STACK_BUFFER_OVERRUN` on Windows) and one for shape mismatches
(candle 0.9 panics internally rather than returning `Err` on bad shapes). Both replaced with
tests that exercise real behaviour.

### Wave 4 — Model Loading and Backend Struct

**File:** `candle_ggqr/backend.rs` (new, ~520 lines, 7 tests)

`GgqrCandleBackend { state: Mutex<Option<LoadedState>> }`.

`LoadedState` holds: `tensors: HashMap<String, Tensor>`, `config: ModelConfig`,
`tokenizer: Tokenizer`, `device: Device`. Everything the forward pass and generation loop need,
all behind one Mutex so the `&self` trait requirement is satisfiable without `unsafe`.

`load_model_impl` sequence:

1. Check ≥ 1 GB RAM free via `sysinfo::System::refresh_memory()`. Hard stop with
   `InsufficientMemory` if not — better to fail loudly here than OOM silently mid-dequantisation.
2. `validate_gguf` — rejects malformed files before allocating anything.
3. `extract_architecture` — confirms the model is supported.
4. `build_model_config` — builds the `ModelConfig`.
5. Loop over all tensors: `dequantize_tensor` → `vec_to_tensor` → insert into `HashMap`.
   Each tensor logs its index, name, dtype, and shape to stderr.
6. Validate that required tensor name patterns are present (`"token_embd"`, `"attn_q"`, etc.).
   Catches half-written model files before attempting inference.
7. `load_tokenizer_from_gguf` — tries `{stem}_tokenizer.json` beside the GGUF, falls back to
   `tokenizer.json` in the same directory. No embedded tokenizer extraction yet
   (GGUF embeds vocab arrays, but the HuggingFace `tokenizers` crate works cleanest from JSON).

`with_state<F, R>` gives callers a closure-based accessor to `LoadedState` without exposing the
type (it's private). Returns `GwenError::InferenceBackend` if no model is loaded.

### Wave 5 — Forward Pass

**File:** `candle_ggqr/forward.rs` (new, ~490 lines, 10 tests)

Four public functions, each independently tested:

**`rms_norm(x, weight, eps)`**

```
y = x / sqrt(mean(x²) + eps) * weight
```

Uses `mean_keepdim` over the last dimension, adds eps as `f64` (candle's add scalar takes f64),
`sqrt`, `broadcast_div`, `broadcast_mul`. The all-zero input test confirms eps prevents division
by zero — RMS stays finite.

**`attention(x, tensors, layer, config)`**

Retrieves `attn_{q,k,v,output}.weight` for the given layer. Applies linear projections with
transposed weights (`x.matmul(&w.t()?)`). Reshapes Q/K/V to `[n_heads, seq_len, head_dim]`,
handles GQA by repeating K/V heads when `n_kv_heads < n_heads`. Computes scaled dot-product
attention: `scores = Q @ K^T / sqrt(head_dim)`, softmax (from `candle_nn::ops`), `context =
weights @ V`. Merges heads back to `[seq_len, hidden]` with `.contiguous().reshape(...)`, then
applies the output projection.

No positional encoding. RoPE is deferred — Qwen3 and LLaMA 3 need it for correct results at
inference time, but the forward pass shape contracts are correct and the tests verify
`[seq, vocab]` output. Adding RoPE is an isolated change to the attention function.

**`mlp(x, tensors, layer)`**

Gate and up projections in parallel (two independent matmuls). `silu(gate) * up` (SwiGLU).
Down projection. Uses `candle_nn::ops::silu` — it is not in `candle_core`.

**`forward(input_ids, tensors, config)`**

Embedding lookup → N transformer layers (pre-attn RMSNorm + attention + residual, pre-MLP
RMSNorm + MLP + residual) → final RMSNorm → lm_head matmul. Returns `[seq_len, vocab_size]`
logits. Input is a 1-D `[seq_len]` u32 tensor of token IDs.

Candle 0.9 note: running tests with `--test-threads=1` is mandatory. Candle installs a global
panic handler that is not thread-safe — parallel tests trigger `STATUS_STACK_BUFFER_OVERRUN` on
Windows. The test suite runs cleanly single-threaded.

### Wave 6 — Token Sampling

**File:** `candle_ggqr/sampling.rs` (new, ~335 lines, 13 tests)

Three functions:

**`greedy_sample(logits)`** — argmax. Uses `.max_by` with `partial_cmp`, falls back to
`Ordering::Equal` for NaN (paranoid, NaN logits should not happen but a silent wrong token is
worse than a random one). Returns `unwrap_or(0)` rather than `.unwrap()` after the non-empty
guard — the guard makes it unreachable, but the code is panic-free by construction.

**`top_p_sample(probs, top_p, rng)`** — sort by probability descending, accumulate until
cumulative mass ≥ `top_p`, re-normalise the nucleus, sample uniformly. The first token that
pushes past the threshold is included so the nucleus always has at least one candidate.
Falls back to the highest-probability token if total nucleus mass is somehow zero (degenerate
input defence).

**`sample_token(logits, params)`** — dispatcher:

- `temperature == 0.0` → greedy, no RNG touch
- `temperature > 0.0` → scale by `1/T`; top-k masking (sets non-top-k entries to `-inf` before
  softmax); numerically stable softmax (subtract max before exp); top-p nucleus if `top_p < 1.0`,
  full categorical otherwise
- `params.seed` → `StdRng::seed_from_u64` for reproducibility; `None` → `from_entropy`

`SmallRng` was the initial choice but requires `rand`'s `small_rng` feature. The crate pins
`rand = "0.8"` without it. `StdRng` is always available and fine for this use case.

`gen` is a reserved keyword in Rust 2024 edition — `rng.r#gen::<f32>()` required.

### Wave 7 — Generation Loop and InferenceBackend Trait Impl

**Files:** `candle_ggqr/generation.rs` (new, ~710 lines, 5 tests),
`candle_ggqr/backend.rs` (extended)

**`generation.rs`** owns three things:

`GenerationState<'a>` — a plain value type (no Mutex, no Arc) that borrows the loaded tensors,
config, tokenizer, and device. Passed into the generation functions directly. The Mutex is
acquired in `backend.rs`, state is borrowed for the generation call, Mutex released. Clean
lifetime discipline — no lock held across await points.

`generate_stream(state, prompt, params)` — `async_stream::stream!` block. Tokenises the prompt
(no special tokens added — caller is responsible for ChatML formatting). Builds a growing
`context: Vec<u32>` starting from the prompt IDs. Each step: build input_ids tensor, run
`forward`, extract the last-position logit row with `.get(last_pos)`, convert to `Vec<f32>`,
call `sample_token`, check EOS IDs (`[2, 128001, 151645, 32000]` covering LLaMA 2, LLaMA 3,
Qwen ChatML, Phi-3), decode with `tokenizer.decode`, check stop sequences in accumulated text,
yield fragment. EOS tokens are not yielded.

Errors inside the stream are encoded as `"[gwen-error] …"` fragments — `Stream<Item=String>`
cannot carry `Result`, and terminating the stream silently on error is worse than surfacing the
message to the caller.

`generate_collect(state, prompt, params)` — drives `generate_stream` to completion via
`tokio::task::block_in_place` + `Handle::current().block_on`. Lifts stream-encoded errors back
to `GwenError::InferenceError`. Requires a multi-threaded Tokio runtime in the caller's context
(which `InferenceBackend::infer` always has — it is called from the chat pipeline that runs
inside `#[tokio::main]`).

`make_stream_pinned(tensors, config, tokenizer, device, prompt, params)` — clones all state into
an owned `stream!` block so the returned `Pin<Box<dyn Stream<Item=String> + Send + 'static>>`
has no borrowed lifetimes. The Mutex is released before this call; the clone is the only copy
of the weights made during streaming. The generation test `make_stream_pinned_is_send_and_terminates`
verifies `Send` by moving the stream to a new thread before collecting.

**`backend.rs` — `InferenceBackend` impl:**

```
name()        → "candle-ggqr"
load_model()  → load_model_impl() + anyhow error bridge
infer()       → with_state() → generate_collect()
stream_infer() → clone LoadedState fields out of Mutex → make_stream_pinned()
unload()      → unload_impl()
```

`stream_infer` clones tensors, config, tokenizer, and device out of the Mutex before constructing
the stream. This is the critical design decision: if the stream held a `MutexGuard`, any call to
`unload()` from another thread would deadlock. Cloning is the correct tradeoff — tensors are
`Arc`-backed internally in Candle, so the clone is reference-count bumps, not weight copies.

The `generate_collect` test used `spawn_blocking` in the first draft. `spawn_blocking` requires
a `'static` closure — `GenerationState` borrows local variables, so it cannot be `'static`.
Replaced with `block_in_place` inside a `block_on` on a multi-thread runtime, which keeps the
borrow on the same thread and satisfies the lifetime.

### Wave 8 — Registry Integration, Selector, and No-Panic Audit

**Files:** `registry.rs`, `selector.rs`, `config.rs`, `sampling.rs`

**Registry** — `BackendRegistry::new()` now registers `GgqrCandleBackend` under `"candle-ggqr"`
when the `candle-backend` feature is active:

```rust
#[cfg(feature = "candle-backend")]
{
    eprintln!("gwen-registry: registering backend 'candle-ggqr'");
    r.register("candle-ggqr", Box::new(GgqrCandleBackend::new()));
}
```

The `list_available()` output is sorted lexicographically. `"candle-ggqr"` sorts before
`"mistralrs"`. Auto selection does not use sort order — it is explicit in `resolve_backend`.

**Selector** — `resolve_backend` now has explicit match arms for `"candle-ggqr"` (canonical)
and `"candle"` (legacy alias that maps to `"candle-ggqr"`). The `"auto"` priority chain is:
mistralrs → candle-ggqr → `ArchitectureNotSupported`. Unknown backend names fall through to the
auto chain rather than hard-erroring, so config files written for future backends degrade
gracefully on older builds.

`select_backend` emits a diagnostic `eprintln!` with the resolved backend name and model path
on every call. Same `eprintln!` convention as the rest of the module — no `log` crate.

**Config** — `InferenceConfig::validate()` whitelist extended with `"candle-ggqr"`. Without this,
`config.toml` with `backend = "candle-ggqr"` would pass selector resolution but fail validation.
The legacy `"candle"` value remains valid.

**No-panic audit** — the grep over all `candle_ggqr/**` showed one production-code `.unwrap()`
in `greedy_sample` after the non-empty guard. Changed to `.unwrap_or(0)`. All other `.unwrap()`
calls are in `#[cfg(test)]` blocks — acceptable per the spec ("no-panic in production code").
The four Mutex `.unwrap()` calls in `backend.rs` are standard Rust — a poisoned Mutex means a
prior thread panicked, which is itself a bug; propagating the panic is correct behaviour.

## Bugs Hit During Implementation

**`Arc<[f32]>` is a DST.** `Arc::try_unwrap` requires `T: Sized`. `Arc<[f32]>` (a slice) is
`!Sized`. Switched to `Arc<Vec<f32>>`. Same ownership semantics, `Sized`, works.

**candle 0.9 panics on 0-element tensors.** `Tensor::from_vec(vec![], ...)` triggers a stack
buffer overrun on Windows. Removed the empty-vec test, documented the candle 0.9 behaviour
in the test file.

**candle 0.9 panics on shape mismatches.** Shape validation is not `Result`-based — the internal
check panics. Replaced the `returns_candle_error` test with a large-reshape smoke test.

**Parallel candle tests crash.** Candle 0.9 has a non-thread-safe global panic handler. Running
more than one candle test concurrently on Windows produces `STATUS_STACK_BUFFER_OVERRUN`. Fixed
by always running the full suite with `--test-threads=1`.

**`log` crate not in scope.** Used `log::info!` and `log::debug!` in early `backend.rs` drafts.
`log` is not a dependency. Replaced all with `eprintln!()`, consistent with `loader.rs`.

**`gen` reserved keyword.** `rng.gen::<f32>()` does not compile in Rust 2024. Required
`rng.r#gen::<f32>()`.

**`SmallRng` needs a cargo feature.** `rand::rngs::SmallRng` requires `features = ["small_rng"]`
in the rand dependency. The crate uses `rand = "0.8"` without that feature. Switched to `StdRng`.

**`spawn_blocking` needs `'static`.** The `generate_collect` test initially used
`rt.block_on(async { spawn_blocking(move || generate_collect(&state, ...)) })`. `spawn_blocking`
moves the closure to another thread, requiring `'static`. `GenerationState` borrows local
variables. Fixed by using `block_in_place` inside `block_on` — same thread, no `'static`
requirement.

**`Tokenizer::from_str` needs `use std::str::FromStr`** in scope. The trait is not in the
prelude. Added the import to the test that injects a tokenizer directly into `LoadedState`.
