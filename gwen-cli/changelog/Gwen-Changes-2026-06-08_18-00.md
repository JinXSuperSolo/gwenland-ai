**Date:** 2026-06-08

**Type:** Feature — trait-based multi-backend inference architecture

## Summary

Implements waves 0–3 of GWEN-211: the foundational layer for integrating
[mistral.rs](https://github.com/EricLBuehler/mistral.rs) as a native Rust
inference backend alongside the existing candle-transformers implementation.

No user-facing behaviour changes. No breaking changes to `chat.rs`, the TUI,
or any public API. The entire mistralrs-backend code path is gated behind
`--features mistralrs-backend` and is dead code in the default build.

Wave 0 adds the build plumbing. Waves 1–3 lay the type system: error variants,
parameter config, the shared trait, the backend registry, GGUF architecture
detection, and the test infrastructure that all later waves depend on.

---

## Why

### The Problem with a Single Backend

GwenLand's inference engine is built on candle-transformers: a pure-Rust ML
framework that is excellent for portability but requires loading weights through
a custom dequant path and driving the generation loop manually. This gives
us full control but means every architecture (LLaMA, Qwen, Phi, Gemma…) needs
a bespoke candle model adapter and sampler implementation.

mistral.rs is also pure Rust, ships its own highly-optimised GGUF loader,
and supports the same model families natively. On RTX 3090 hardware it can
sustain >90 tokens/s for Qwen3-1.7B Q8_0. Adding it as an alternative backend
allows users to choose the right tradeoff (portability vs throughput) via a
single config field without recompiling.

### Why a Trait, Not a Conditional Import

The naive approach — `#[cfg(feature = "mistralrs-backend")] use mistralrs` at
every call site — would scatter feature guards throughout `chat.rs`, `serve.rs`,
and the benchmark harness. It also makes testing impossible without the feature
flag active, because there is no stable interface to mock.

A trait (`InferenceBackend`) lets the call sites stay completely free of feature
guards. `chat.rs` talks to `Arc<dyn InferenceBackend>` and never knows whether
it is driving candle or mistralrs. The registry (`BackendRegistry`) is the only
place that gates on feature flags, and the mock (`MockBackend`) lets every test
run without any backend compiled in.

### Why Now (Wave 0–3 First)

Shipping the mistralrs crate in one massive PR would make bisecting regressions
nearly impossible. The wave structure in tasks.md mirrors this: build plumbing
first, types second, trait third, infrastructure third — so that each wave can
be reviewed and verified independently before the actual mistralrs API calls
land in wave 4 (task 8.1).

Waves 0–3 are pure Rust, introduce zero runtime behaviour change, and have a
full test suite. They are safe to merge even before mistralrs integration is
complete.

### Why the GGUF Architecture Detection Belongs Here

The existing `gguf_parser.rs` skips all KV metadata — it only needs tensor
offsets for dequantisation. `detect_architecture()` needs the opposite: it
reads only the KV section (specifically `general.architecture`) and ignores
tensors entirely. This is a fast, header-only operation that works on any GGUF
file regardless of size. Putting it in `arch_detect.rs` (rather than bloating
`gguf_parser.rs` with dual-purpose logic) keeps the two concerns clean. It also
reuses the already-public primitive readers (`read_u32_le`, `read_u64_le`,
`read_gguf_string`) from `gguf_parser.rs` to avoid duplicating low-level I/O
code.

### Why Strict Security Limits in arch_detect

The GGUF spec allows a file to claim an arbitrary `tensor_count` in its header.
A 4-byte field can claim 4 billion tensors. The existing `gguf_parser.rs` would
allocate a `Vec` of that size if it ever tried to iterate it. `arch_detect.rs`
reads `tensor_count` and hard-rejects any file claiming more than 10,000 — this
is a header-only read, happens before any allocation, and addresses Requirement
16.3. The 10,000 limit is conservative: the largest production models (e.g.
Llama-3 70B) have ~559 tensors.

### Wave 1a — `src/error.rs` (new file)

New top-level error type for gwenland-core, wired into `src/lib.rs`:

```rust
use thiserror::Error;

#[derive(Debug, Error)]
pub enum GwenError {
    #[error("inference backend error: {0}")]
    InferenceBackend(String),

    #[error("backend '{backend}' not available (compile with --features {backend}-backend)")]
    BackendNotAvailable { backend: String },

    #[error("model architecture not supported by backend '{backend}': {arch}")]
    UnsupportedArchitecture { backend: String, arch: String },

    #[error("model loading failed: {0}")]
    ModelLoad(String),
}
```

`lib.rs` changes:
```rust
pub mod error;
pub use error::GwenError;
```

All four variants carry the identifying names (backend, arch) directly in the
error message so that `Display` output is self-contained — no need to unwrap
the struct fields to know what failed.

### Wave 2 — `src/engine/inference/backend.rs` (new file)

The core abstraction. All call sites downstream of this file are entirely
backend-agnostic:

```rust
pub trait InferenceBackend: Send + Sync {
    fn load_model(&self, model_path: &Path) -> Result<()>;
    fn infer(&self, prompt: &str, params: &InferParams) -> Result<String>;
    fn stream_infer(&self, prompt: &str, params: &InferParams)
        -> Result<Pin<Box<dyn Stream<Item = String> + Send>>>;
    fn unload(&self) -> Result<()>;
    fn name(&self) -> &'static str;
}
```

Design decisions:
- **`&self` on all methods** — interior mutability (`Arc<Mutex<...>>`) is the
  implementer's responsibility. This keeps the trait `Send + Sync` without
  any `unsafe` and allows `Arc<dyn InferenceBackend>` to be cloned freely
  across async tasks.
- **`stream_infer` returns `Pin<Box<dyn Stream<...> + Send>>`** — avoids naming
  the concrete future type. The `Send` bound lets the stream cross thread
  boundaries in the tokio runtime.
- **`futures_util::Stream`** — uses the already-direct dep rather than adding
  `futures-core` separately.

Re-exported from `engine/inference/mod.rs` as `pub use backend::InferenceBackend`.

### Wave 3b — `src/engine/inference/arch_detect.rs` (new file)

Header-only GGUF architecture detection for the mistralrs backend. Reads only
the KV section — no tensor data is loaded:

```rust
pub fn detect_architecture(path: &Path) -> Result<&'static str>
```

**Processing pipeline:**
1. Open file and read magic u32 → reject if ≠ `0x4647_4755` (Req 16.2)
2. Read version u32 (accepted: 1–3)
3. Read `tensor_count` u64 → reject if > 10,000 (Req 16.3)
4. Read `kv_count` u64
5. Scan KV entries, skipping non-target keys, until `general.architecture` found
6. Map architecture string to mistralrs identifier:

| GGUF value | Returns |
|---|---|
| `"llama"` | `"llama"` |
| `"qwen"` or `"qwen2"` | `"qwen2"` |
| `"phi3"` | `"phi3"` |
| `"mistral"` | `"mistral"` |
| `"gemma"` | `"gemma"` |
| anything else | `Err` containing the unsupported name |

Reuses `read_u32_le`, `read_u64_le`, `read_gguf_string` from
`crate::convert::gguf_parser` (all `pub`). Adds its own `skip_kv_value` that
handles all 13 GGUF KV type codes (including arrays — recursive skip) and
`read_byte` / `read_exact` helpers.

**14 unit tests** using in-memory GGUF files built with `NamedTempFile`:
- All 6 supported arch mappings (including both `qwen` and `qwen2`)
- Unsupported arch (`"rwkv"`) returns `Err` containing the arch name
- Invalid magic bytes → `Err`
- `tensor_count = 10,001` → `Err` (security limit)
- `tensor_count = 10,000` → accepted (boundary)
- Missing `general.architecture` key → `Err`
- Arch key not in first position (skip logic exercised)
- Numeric KV types (UINT32, FLOAT32, UINT64) before arch key (skip coverage)
- Nonexistent file → `Err`

### Wave 3c — `src/engine/inference/mock_backend.rs` (new file)

Test double for `InferenceBackend`. Only compiled with `#[cfg(any(test, feature = "test-utils"))]`:

```rust
pub struct MockBackend {
    inner: Arc<Mutex<MockBackendInner>>,
}

struct MockBackendInner {
    tokens: Vec<String>,
    fail_on_load: bool,
    load_count: usize,
    unload_count: usize,
    loaded: bool,
}
```

**Constructors:**
- `MockBackend::new(tokens: Vec<String>)` — succeeds on load, yields tokens
- `MockBackend::with_fail_on_load(tokens: Vec<String>)` — load_model returns Err

**Observable state accessors:**
- `load_count()` — number of times `load_model` was called successfully
- `unload_count()` — number of times `unload` was called
- `is_loaded()` — current loaded state

**Trait implementation:**
- `load_model` — increments counter, sets `loaded = true` (or fails)
- `infer` — returns `tokens[..max_tokens].join("")`
- `stream_infer` — yields tokens via `async_stream::stream!`, stops at
  `max_tokens` or when a stop sequence is encountered
- `unload` — increments counter, sets `loaded = false`
- `name` — returns `"mock"`

Used by property tests in waves 4–9 and for the chat integration tests in wave 8.
