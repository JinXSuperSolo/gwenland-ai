## Executive Summary

Waves 6–8 complete the production-facing layer of GWEN-211. The foundation
(Waves 0–5) gave us the trait, the config struct, and the mistralrs backend
implementation. These three waves turn that foundation into a working end-to-end
path: a function that picks the right backend and resolves the model path (Wave
6), a registry that actually registers mistralrs at startup (Wave 7), and a
native inference path wired directly into `chat.rs` that bypasses the SSE proxy
entirely (Wave 8).

After these waves, a user who compiles with `--features mistralrs-backend` and
sets `[inference]` in their config can run `gwen chat` and have the model
inference happen entirely inside the Rust process — no HTTP, no proxy, no Ollama
dependency.

## Why

### Wave 6 — Why a Dedicated `select_backend` Function?

Before Wave 6, there was no single place that answered the two questions every
inference call site needs:

1. Which backend name should I pass to the registry?
2. What is the absolute, fully-expanded path to the model file?

Without `select_backend`, every call site would have had to re-implement tilde
expansion, `.gguf` validation, the feature-flag check, and the "auto" priority
logic. Spreading that across `chat.rs`, `serve.rs`, and any future CLI command
would create three separate bugs whenever the rules changed.

**Why not put the logic inside `InferenceConfig::validate()`?**
`validate()` checks that the *config is internally consistent* — it is called
at parse time and should not make runtime decisions about which compiled-in
features are available. `select_backend` is a *runtime policy* function: it
reads the config and the compiled feature set together and produces an
actionable (backend, path) pair. Mixing those concerns into `validate()` would
make testing harder (you would need a fully populated `InferenceConfig` with a
real `model_path` directory just to test the backend selection logic).

**Why return `(&'static str, PathBuf)` rather than an `Arc<dyn InferenceBackend>`?**
`select_backend` does not own a registry instance and should not. Its job is to
answer *which* backend and *where* the file is. The caller (chat, serve, CLI)
already holds the registry and can do the `registry.get(name)` lookup itself.
Returning a name string keeps the function pure and trivially testable without
constructing any backend objects.

**Why `GwenError` instead of `anyhow::Error`?**
The three failure modes (`no model_path set`, `non-.gguf extension`,
`BackendNotAvailable`) are all *structured* errors that the CLI needs to match
on in order to give the user a specific fix hint. `anyhow::Error` erases the
variant and makes matching impossible. `GwenError` carries the variant so the
CLI can say "compile with `--features mistralrs-backend`" rather than a generic
error string.

### Wave 7 — Why Wire the Registry Here and Not Earlier?

`BackendRegistry::new()` was intentionally left empty in Waves 0–3. The
registry existed to hold backends, but no backends were registered because
`MistralRsBackend` was not yet implemented (it landed in Wave 4–5). Wiring
an incomplete implementation into `new()` would have caused compilation errors
under `--features mistralrs-backend` at every wave boundary, making incremental
review impossible.

Wave 7 is the first moment when `MistralRsBackend::new()` is stable and the
`select_backend` function exists to actually use what the registry returns.
Wiring them together at this point is the natural completion of the "registry
owns backends" contract established in Wave 3.

**Why `#[allow(unused_mut)]` on the `let mut r` binding?**
Without the `mistralrs-backend` feature, the `r.register(...)` call is compiled
out, leaving `r` as an immutable binding that never needs `mut`. The compiler
correctly warns about the unnecessary `mut`. Rather than restructuring the code
with a nested `if cfg!(...)` block (which would make the feature-gated
registration less readable), the `#[allow(unused_mut)]` attribute suppresses
the warning at the one line where it is expected and documented. The attribute
is local — it does not suppress anything else.

**Why not also register `candle` in Wave 7?**
The existing candle path does not implement the `InferenceBackend` trait yet —
it drives generation through the bespoke `runner.rs` / `sampler.rs` pipeline
rather than through a trait object. Wiring candle into the registry would
require implementing the trait on the existing runner, which is Wave 9+ scope.
Registering only mistralrs here keeps the diff minimal and the registry honest:
`list_available()` returns only backends that are actually usable.

### Wave 8 — Why a Native Path in `chat.rs` Rather Than a Separate Module?

The existing `chat.rs` already owns `ChatSession`, `ChatEvent`, `ChatMessage`,
and the SSE streaming logic. It is the right home for a second streaming path
because:

- The `ChatSession` mutation (`finalize_assistant_turn`) must happen in the
  same function that consumes the stream, regardless of which backend produced
  the tokens.
- `HeadlessChatConfig` is already the single struct that parameterises a
  non-TUI chat run. Adding `native_backend` and `native_params` fields there
  is the minimum additive change that avoids a new config type and a new
  dispatch site in the CLI.
- `stream_chat_native` follows the exact same `(session, backend, params, tx)`
  call pattern as `stream_chat`, so the TUI can call either function with the
  same surrounding code.

**Why not feature-gate `stream_chat_native`?**
The function takes `Arc<dyn InferenceBackend>`, which is always compiled in
regardless of features (it is part of the trait definition in `backend.rs`,
which has no feature gate). The caller already decides whether to construct a
real `MistralRsBackend` or a `MockBackend` — `stream_chat_native` itself does
not need to know. Leaving it always-compiled means tests can use it with
`MockBackend` without needing `--features mistralrs-backend`, which is exactly
the contract the test suite relies on.

**Why the `<|system|>/<|user|>/<|assistant|>` chat template?**
This is the instruction-following prompt format used by the three primary target
model families: Qwen3, LLaMA 3 Instruct, and Mistral Instruct. It is not the
only format (ChatML uses `<|im_start|>`/`<|im_end|>`, Phi-3 uses `<|user|>`
with different wrapping) but it is the common denominator that works without
modification on all three. A per-model template system is Wave 9+ scope. Using
the shared format here means the native path works for the most common models
immediately, without a tokenizer configuration step.

**Why does `build_prompt_from_session` always append `<|assistant|>\n` at the end?**
This is the generation primer — it tells the model that it is the assistant's
turn to respond. Without it, models that have been trained on this template
format will produce a `<|assistant|>` token as their first output (wasting one
token and sometimes causing a newline in the output). Appending it explicitly
removes the model's need to produce it and results in cleaner token-by-token
streaming.

**Why keep the SSE path entirely unchanged and just add a native branch at the top of `run_headless_chat`?**
The hard constraint for this wave is zero breaking changes. Any restructuring of
the existing SSE code risks introducing a regression in a code path that the TUI
relies on. The `if let (Some(backend), Some(params)) = ...` guard at the top is
the minimum possible change: it is an early return that the existing code never
reaches when both fields are `None` (the default). The SSE path literally does
not know the guard exists.

### Wave 6 — `selector.rs` (NEW, ~140 lines, 12 tests)

**File:** `src/engine/inference/selector.rs`

**Public API:**
```rust
pub fn select_backend(config: &InferenceConfig) -> Result<(&'static str, PathBuf), GwenError>
```

**Decision table:**

| `model_path` | Extension | `backend` field | Feature | Returns |
|---|---|---|---|---|
| empty `""` | — | — | — | `Err(InferenceBackend("no model_path set"))` |
| any | not `.gguf` | — | — | `Err(ModelLoad(...))` |
| valid `.gguf` | `.gguf` | `"mistralrs"` | absent | `Err(BackendNotAvailable)` |
| valid `.gguf` | `.gguf` | `"mistralrs"` | present | `Ok(("mistralrs", path))` |
| valid `.gguf` | `.gguf` | `"auto"` | absent | `Ok(("candle", path))` |
| valid `.gguf` | `.gguf` | `"auto"` | present | `Ok(("mistralrs", path))` |
| valid `.gguf` | `.gguf` | `"candle"` | any | `Ok(("candle", path))` |

**Tilde expansion:** `expand_tilde()` replaces a leading `~/` with
`dirs::home_dir()`. If `dirs::home_dir()` returns `None` (no `$HOME` set, as
in some CI environments), the path is returned unchanged — the function is
correct but the subsequent model load will fail with a path-not-found error.

**`mod.rs` changes:**
```rust

```

### Wave 7 — `registry.rs` (MODIFIED)

**Change:** `BackendRegistry::new()` now registers `MistralRsBackend` under
`"mistralrs"` when the feature is compiled in.

```rust
#[allow(unused_mut)]
let mut r = Self { backends: HashMap::new() };
#[cfg(feature = "mistralrs-backend")]
r.register("mistralrs", Box::new(MistralRsBackend::new()));
r
```

**Import added:**
```rust
#[cfg(feature = "mistralrs-backend")]
use super::mistralrs_backend::MistralRsBackend;
```

The import is itself feature-gated, so `MistralRsBackend` is never in scope in
the default build. This satisfies the zero-link-overhead constraint from Wave 0.

### Wave 8 — `chat.rs` (MODIFIED)

#### New fields on `HeadlessChatConfig`

```rust
pub native_backend: Option<Arc<dyn crate::engine::inference::InferenceBackend>>,
pub native_params: Option<crate::engine::inference::InferParams>,
```

Both default to `None` when constructed without them. All existing call sites
that construct `HeadlessChatConfig` by name need these two new fields, but since
Rust struct construction is exhaustive, the compiler would have flagged any
missing site — none exist outside the test module.

#### `build_prompt_from_session` (private)

Formats the session history using the `<|system|>/<|user|>/<|assistant|>`
template. Output always ends with `<|assistant|>\n` to prime generation.

#### `stream_chat_native` (public, always compiled)

```rust
pub async fn stream_chat_native(
    session: &mut ChatSession,
    backend: Arc<dyn crate::engine::inference::InferenceBackend>,
    params: &crate::engine::inference::InferParams,
    tx: UnboundedSender<ChatEvent>,
)
```

Calls `backend.stream_infer(&prompt, params)`, iterates the returned async
stream with `.next().await`, forwards each token as `ChatEvent::Token`, commits
the accumulated response via `session.finalize_assistant_turn`, then sends
`ChatEvent::Done`. On `stream_infer` error, sends `ChatEvent::Error` and
returns without mutating the session.

#### `run_headless_chat` early-return native path

```rust
if let (Some(backend), Some(params)) = (&cfg.native_backend, &cfg.native_params) {
    // ... native path ...
    return EXIT_OK;
}
// existing SSE path below — untouched
```

The guard pattern requires *both* fields to be `Some`. If only one is set (user
error), the SSE path runs as before, which is the safest degradation.
