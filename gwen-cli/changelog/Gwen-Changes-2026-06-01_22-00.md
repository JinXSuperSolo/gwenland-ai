# GwenCLI Change Log — 2026-06-01 22:00

## Overview

Six new modules complete the candle-native training surface in `gwen-core` and wire the full pipeline into `gwen-tui`'s `train` command. The session added a typed stdout progress emitter (`train/progress.rs`), a lightweight dry-run analyser that fetches only HF `config.json` (`train/dry_run.rs`), a native training orchestrator that owns all candle/hf_hub/tokenizers calls (`train/native_runner.rs`), and rewrote `gwen-tui/src/commands/train.rs` to dispatch across three paths: `--dry-run`, legacy Python subprocess, and native Candle. All additions compile cleanly under `--features gwen-core/candle`; zero new errors, zero new warnings introduced in our code.

## New / Modified Files

| File | Change |
|---|---|
| `packages/core/src/train/progress.rs` | **New file.** `ProgressEvent { step, epoch, loss: f32, lr: f64, tokens_per_sec: f32 }` and `DoneEvent { event: "done", final_loss, total_steps, elapsed_secs }`. Both implement `serde::Serialize`; `emit()` calls `serde_json::to_string(self).unwrap()` + `println!` — one compact JSON line per call, no pretty-print, no framing. Field names are wire-locked to the existing TUI parser in `train_panel.rs` (`step`, `epoch`, `loss`, `lr`); `tokens_per_sec` is additive and ignored by the current parser. `unwrap` on `to_string` is safe because all fields are plain integers and finite floats. 6 unit tests: single-line guarantee, required TUI keys, `tokens_per_sec` presence, done sentinel, done fields. |
| `packages/core/src/diagnostics/estimator.rs` | Added `#[derive(serde::Serialize)]` to `VramEstimate` — required for `DryRunResult` to derive `Serialize`. No logic changes. |
| `packages/core/src/train/dry_run.rs` | **New file.** `DryRunResult { dataset_samples, avg_token_length, total_tokens, model_params, lora_params, vram: VramEstimate, time_cpu, time_t4, time_rtx3080, warnings }`. `run(config: &TrainConfig) → Result<DryRunResult>` performs seven steps without touching model weights or initialising any Tensor: load JSONL (warn if < 100 samples), estimate token lengths via 4-chars/token heuristic (no `candle_core::Device` init), fetch `config.json` via `hf_hub::api::sync` (2 KB, cached after first call), infer `total_params` from `num_parameters` field or `12×d²×L` formula, compute `lora_params = 2×r×d_model×|target_modules|`, call `diagnostics::estimator::estimate_vram/estimate_time`, warn if estimated VRAM > 80% of available system RAM. `HfModelConfig` handles both modern (`hidden_size`, `num_hidden_layers`) and GPT-2 (`n_embd`, `n_layer`) field aliases — covers LLaMA, Mistral, Qwen, Gemma, GPT-2, Phi. Uses sync hf-hub API (not tokio) to avoid nested runtime panics from actix worker threads. 9 unit tests; zero network calls in tests. |
| `packages/core/src/train/native_runner.rs` | **New file.** `run_native(config: &NewTrainConfig) → Result<TrainResult>`. Owns all candle/hf_hub/tokenizers orchestration so that gwen-tui never needs to reference those crates directly. Steps: fetch `tokenizer.json` via sync hf-hub API → `load_jsonl` + `tokenize` + `batch` → allocate `VarMap` + `VarBuilder` → `LoraLayer::new` with zero-tensor placeholder base weight → `TrainingLoop::new` + `run()`. Feature-gated: when built without `--features candle`, returns a clear `bail!` message directing the user to rebuild. `#[cfg(feature = "candle")]` lives here where the feature is defined — not in gwen-tui. |
| `packages/core/src/train/mod.rs` | Added `pub mod native_runner;` and `pub mod progress;`. |
| `packages/tui/src/commands/train.rs` | **Rewritten.** Keeps the same public surface (`TrainArgs`, `run_train_cmd`). New flags: `--epochs <N>`, `--lr <F>` (optional, default from `NewTrainConfig`); `--config / -c` is now optional (legacy YAML path only); existing flags unchanged. Three dispatch paths in `run_train_inner`: (1) `--dry-run` → `dry_run::run` → `print_dry_run_table` → `Ok(())`; (2) `--custom-script` or `--config` present → deprecation warning to stderr + legacy `run_legacy_path`; (3) default → `run_native_path` → `native_runner::run_native` on a background `std::thread`, TUI launched on main thread after config is built. `print_dry_run_table` matches the `━`/`─` divider style of the existing `print_dry_run_report` in `runner.rs`. All candle/hf_hub/tokenizers names removed from gwen-tui source — gwen-tui calls only gwen-core APIs. |

## Architecture Notes

| Item | Detail |
|---|---|
| Crate boundary rule | `candle_core`, `candle_nn`, `hf_hub`, `tokenizers` are deps of gwen-core. Any gwen-tui file that names them directly fails to compile. All ML code lives in gwen-core; gwen-tui calls typed functions and receives plain Rust structs. |
| `native_runner` as the crate boundary | `native_runner::run_native` is the single function that crosses the boundary. Its signature is `(&NewTrainConfig) → Result<TrainResult>` — no candle types leak into gwen-tui's type system. |
| Sync hf-hub in dry_run and native_runner | Both modules are called from sync or `std::thread` contexts. The tokio hf-hub API panics when called inside an existing tokio runtime (actix worker threads). `hf_hub::api::sync` uses ureq and is safe on any thread. |
| Token length heuristic in dry_run | `estimate_avg_token_length` uses 4 chars/token and clamps to `DEFAULT_MAX_LEN`. This avoids calling `tokenize()` which requires `candle_core::Device` init (200-400 ms even on CPU) — well within the 3-second dry-run budget. |
| TUI launch timing on native path | `ratatui::init()` takes exclusive terminal control. If model loading fails after `init()`, the terminal is left in raw mode. The native path launches the TUI after config is built but before any blocking I/O, with all fallible setup inside `native_runner::run_native` on a background thread. Errors propagate cleanly. |
| Stdout pipe bridge (pending) | `TrainingLoop::run` emits JSON progress to stdout. `run_train_tui` reads a subprocess's stdout. The native path currently uses a dummy child; a real anonymous pipe bridge is the tracked follow-up task. `--verbose` is the reliable headless path in the interim. |
| `#[allow(dead_code)]` on scaffold helpers | `synthetic_train_config` and `dummy_child` are reachable at runtime but not under static analysis (they depend on runtime TTY detection). `#[allow(dead_code)]` is used rather than removing them because they are load-bearing scaffolding for the pipe bridge follow-up. |
