# GwenLand Change Log — 2026-06-04 23:30

## Overview

Cycle 4 — **Settings screen**. Built the full Settings page (`active === 'settings'`)
on top of the existing Dashboard + Chat + Model Manager. This is the first
screen to persist user state to disk, so it also wires the Tauri **fs**,
**shell**, and **dialog** plugins (JS packages + Rust crates + scoped
capabilities) into the previously plugin-free app.

1. **Settings page** — Left sub-nav (General / Proxy / Paths / Shortcuts /
   About / Danger) + scrollable content pane. Six tabs, an inline "Saved"
   toast, and a loading skeleton.

2. **Config persistence** — `useSettings` reads/writes
   `~/.config/gwen/config.json` via the fs plugin, merging over
   `DEFAULT_CONFIG` and debouncing writes 500 ms. Falls back to defaults when
   the file is missing or malformed.

3. **Tauri plugin wire** — Added fs/shell/dialog to the JS deps and the Rust
   crate, registered all three in `lib.rs`, and granted **scoped** permissions
   in `capabilities/default.json` (fs limited to `$HOME/.config/gwen/`, shell
   `open` to GitHub + the config dir, shell `execute` to exactly
   `gwen update --check`).

4. **Hand-rolled controls** — Toggle, input, select, and path-row primitives,
   all inline-styled against the existing Gwen Noir CSS-variable palette (no
   UI library, consistent with the rest of the GUI).

## Architecture Notes

| Item | Detail |
|---|---|
| Persistence target | Config lives at `~/.config/gwen/config.json`, shared with the Rust core and CLI. The GUI is one of several writers, so the file on disk is the source of truth — no localStorage (which would silo the GUI copy and drift). |
| Debounced writes | `update()` coalesces rapid edits (e.g. per-keystroke port typing) into one fs write 500 ms after the last change. Avoids disk races and wasted writes. |
| Defaults merge | Load does `{ ...DEFAULT_CONFIG, ...JSON.parse(raw) }` so partial/older files yield a complete config with no `undefined` fields. Malformed/missing → silent fallback to defaults; the core owns initial file creation. |
| Reset preserves paths | `reset()` rewrites DEFAULT_CONFIG but carries over `configDir`/`modelsDir`/`sessionsDir`, which are core-owned and must survive a settings reset. |
| Inline-style convention | All controls use inline styles + the existing `--bg`/`--orange`/`--text-*`/`--border`/`--radius-*` tokens rather than Tailwind utility classes, matching every other screen in the GUI. The spec's oklch/Tailwind values were mapped onto the existing palette. |
| Double-click confirm | DangerZone arms on first click ("Confirm? Click again") with a 3 s timer; a second click executes, otherwise it reverts. No modal/dialog, per spec. Failed deletes leave files in place (safe outcome). |
| Capability scoping | fs permissions are path-scoped to `$HOME/.config/gwen/`; shell `execute` is locked to the exact `gwen update --check` argv; shell `open` is limited to GitHub URLs and the config dir. No broad filesystem or command access granted. |
| Number coercion | InputSetting stays string-typed; numeric fields coerce at the update site (`Number(v) || 0`) so an empty mid-edit field never produces NaN in the persisted config. |
| First plugin wire | The GUI was previously plugin-free (HTTP-only to Ollama/proxy). This is the first screen needing local OS access; the three plugins are registered once and reusable by future screens. |

## Notes / Deviations

- **Stack reality vs. spec.** The spec described Tailwind v3 utility classes and
  assumed the fs/shell/dialog plugins were already present. The actual GUI uses
  Tailwind v4 (CSS-first) with **inline styles + CSS variables**, and had **zero**
  Tauri plugins. Implemented to the codebase's real conventions; added the three
  plugins (user-approved) rather than installing form libraries or rewriting the
  styling system.
- **Browse mode** on `PathSetting` is implemented (dialog.open) but unused this
  cycle — all path rows are read-only `mode="open"`, since the directories are
  core-owned. Wiring is in place for future editable paths.
- The `gwen update --check` and Clear Sessions paths are code-complete but can't
  be live-verified here: the `gwen` CLI binary isn't published yet (postinstall
  download returns 404), and no populated `~/.config/gwen/` exists in this env.
