## Executive Summary

Three independent threads landed this session, plus one diagnosis:

1. **GitLab CI/CD** ‚Äî a pipeline that checks/tests/builds the Rust workspace and the GUI frontend,
   with optional desktop-bundle jobs.
2. **The GUI wasn't displaying** because the entire frontend build tooling (`package.json`,
   `vite.config.ts`, `tsconfig*`) was missing/untracked ‚Äî the Tauri webview had no built `dist/` to
   load. Restored it, plus fixed a broken bundle icon reference and wired the `gwen` CLI into the
   installer as a Tauri **sidecar**.
3. **`gwen serve <MODEL>` rejected the model** when users followed the app's own hints, which told
   them to use a `--model` flag that didn't exist. Fixed the contract and the hints.
4. **Diagnosed (not fixed):** `gwen serve` can hang for 10+ minutes on first message ‚Äî the proxy
   fetches the tokenizer from HuggingFace Hub over the network, keyed on the local model name.

## 1. GitLab CI/CD (`.gitlab-ci.yml`, new)

**Issue:** No CI existed. The Cargo workspace root is `gwen-cli/` (not the repo root), and the GUI
crate can't be `cargo check`ed standalone (its `tauri::generate_context!` needs a built frontend +
icons at compile time).

**Change:** Three stages ‚Äî `check` / `test` / `build`:
- `check`: `cargo check -p gwenland-core -p gwenland-tui --locked` (gui excluded by design),
  `fmt` + `clippy` (allow-failure), and a frontend `gui:typecheck`.
- `test`: `cargo test -p gwenland-core --lib -- --test-threads=1` (process-global `panic::set_hook`
  / `GWEN_HOME` test env force single-threaded) and `cargo test -p gwenland-tui`.
- `build`: release `gwenland` CLI (default-branch/tags only ‚Äî fat LTO is slow), the GUI `dist/`,
  and an optional manual `bundle:linux` (WebKitGTK) + a Windows bundle template.

All Rust jobs `cd gwen-cli` first; a `.node`/`.rust` job template carries the toolchain + cache.

## 2. The GUI didn't display ‚Äî root cause and fix

**Issue / root cause:** `packages/gui/` had React source and `index.html` but **no build tooling at
all** ‚Äî `package.json`, `vite.config.ts`, and `tsconfig*` were absent (and untracked). `tauri.conf.json`
drives the frontend with `pnpm dev` / `pnpm build` and expects a Vite dev server on `:1420` and a built
bundle at `../dist`. With none of that present, `tauri dev`/`build` had nothing to serve ‚Üí the webview
rendered blank. The CSS entry also uses **Tailwind v4** (`@import "tailwindcss"`), which needs the
`@tailwindcss/vite` plugin to resolve.

**Fix:**
- Added `package.json` (React 18, lucide-react, the Tauri v2 JS packages, Tailwind v4 +
  `@tailwindcss/vite`, Vite 6, TypeScript), `vite.config.ts` (port 1420 / `strictPort`, `outDir: dist`,
  `envPrefix` exposing `TAURI_*`, React + Tailwind plugins), and `tsconfig.json` / `tsconfig.node.json`.
- Verified: `pnpm install` ‚Üí `pnpm build` emits `dist/` (1627 modules), `pnpm typecheck` clean.

### 2a. Executable icon

**Issue:** `tauri.conf.json` `bundle.icon` listed `icons/icon.png`, which had been **deleted** from the
repo. A missing icon source breaks the bundle. The user supplied a new `icons/icon.ico` to use as the
executable icon.

**Fix:** Pointed `bundle.icon` at the files that actually exist ‚Äî
`["icons/32x32.png", "icons/icons.png", "icons/icon.ico"]` ‚Äî keeping `icon.ico` as the Windows exe
icon. Verified post-build: the embedded exe icon matches `icon.ico` (32√ó32). (Per the project's
"no bloat assets" rule, no multi-resolution icon set was generated.)

### 2b. CLI shipped inside the installer (Tauri sidecar)

**Issue:** The GUI shells out to the `gwen` CLI (`App.tsx` auto-starts `gwen serve`; `AboutCard.tsx`
runs `gwen update --check`), but the installer shipped only the GUI ‚Äî so on an installed machine the
CLI wasn't guaranteed to be on `PATH`. The auto-start also used `Command.create('gwen', ‚Ä¶)` against a
shell scope that only whitelisted `update --check`, and **`shell:allow-spawn` was missing entirely**,
so `gwen serve` was silently denied even when a CLI was present.

**Fix:**
- `bundle.externalBin: ["binaries/gwen"]` + `beforeBuildCommand: "pnpm build && pnpm sidecar"`.
- New cross-platform `scripts/prepare-sidecar.mjs`: builds `gwenland-tui` and copies it to
  `src-tauri/binaries/gwen-<target-triple>(.exe)` (detected via `rustc -vV`). `/binaries` is gitignored.
- Frontend `Command.create('gwen', ‚Ä¶)` ‚Üí `Command.sidecar('binaries/gwen', ‚Ä¶)` in both call sites.
- `capabilities/default.json`: sidecar-scoped `shell:allow-execute` (`update --check`) **and** added
  `shell:allow-spawn` (`serve`) ‚Äî fixing the long-silent auto-start denial.
- Verified: full `tauri build` produced `GwenLand_0.1.0_x64_en-US.msi` and `..._x64-setup.exe`, with
  `gwen-x86_64-pc-windows-msvc.exe` bundled.

## 3. `gwen serve` rejected the model ‚Äî root cause and fix

**Issue (reported):** `gwenland.exe serve -- --model "C:\‚Ä¶\blobs\sha256-‚Ä¶"` ‚Üí
`error: unexpected argument 'C:\‚Ä¶' found`. `gwen chat` worked.

**Root cause:** `serve` defines the model as a **required positional** `<MODEL>` ‚Äî there is no
`--model` flag. But the app's own hints told users to run `gwen serve --model <‚Ä¶>`:
- `tui/commands/chat.rs` "inference server down" hint
- `core/engine/stream.rs` (duplicate constant)
- `core/engine/chat.rs` `server_unreachable_message()` ‚Äî the "‚öÝ No server running" banner shown in
  the TUI/GUI

The inconsistency arose because `gwen fetch` and `gwen train` *do* use `-m/--model`. Passing
`--model X` to `serve` makes clap see `--model` as an unknown option; with the user's extra `--`
(end-of-options), `--model` is taken as the single positional and the real path becomes a second,
unexpected positional. `gwen chat` "works" because it takes no model ‚Äî it just connects to the proxy.

**Fix (both directions):**
- **Contract (B):** `serve` now accepts the model three ways ‚Äî positional `<MODEL>` (unchanged),
  `-m/--model <MODEL>` (new; `conflicts_with` the positional), or **bare `gwen serve`** which falls
  back to the last-served model via the new `platform::serve::read_last_used_model()`
  (wrapping `storage::config::read_last_used_model`). The bare-`serve` fallback also fixes the GUI
  auto-start, which spawns `gwen serve` with no model.
- **Hints (A):** the three strings now show the canonical positional form `gwen serve <path/to/model.gguf>`.
- Verified on the rebuilt binary: `serve --help` shows `[MODEL]` + `-m, --model`, and
  `gwen serve --model <bogus>` now reaches model resolution instead of a clap parse error.

> Note: `gwen serve -- --model "<path>"` remains invalid ‚Äî `--` is the standard end-of-options
> marker. The corrected hints don't include `--`, so the forms users will copy now work.

## 4. Diagnosed (NOT fixed): `gwen serve` 10-minute load "ghost bug"

**Symptom:** after `serve`, the model "isn't served" for 10+ minutes on the first chat message.

**Root cause:** the proxy loads nothing at startup; it calls `run_inference` **per chat request**
(`platform/proxy.rs`). The first thing `run_inference` does ‚Äî before the GGUF ‚Äî is fetch the tokenizer
**over the network from HuggingFace Hub** (`engine/inference/loader.rs::load_tokenizer`), keyed on
`model_id_for_tokenizer`, which the proxy sets to `body.model` (the local model name / Ollama blob hash).
That is not a real HF repo id, so the sync hf-hub request (no timeout, `with_progress(true)`, retries)
**blocks for minutes** instead of failing fast ‚Äî before the model is ever loaded. Aggravating factors:
no model/tokenizer caching (full 4.4 GB reload every message), and a working *local* tokenizer loader
(`engine/inference/candle_ggqr/backend.rs::load_tokenizer_from_gguf`) that this path ignores.

**Proposed fix (pending approval):** (1) load the tokenizer locally from the GGUF/sidecar instead of
HF Hub (reuse `load_tokenizer_from_gguf`); make any HF use opt-in with a short timeout. (2) Cache the
loaded model+tokenizer in the proxy `AppState`, keyed by model path, so it loads once, not per request.

*(Separate, also unfixed: the GUI posts to `/chat` while the proxy serves `/gwenland/chat` ‚Äî a path
mismatch to verify.)*
