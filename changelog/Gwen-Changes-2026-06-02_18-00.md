# GwenCLI Change Log — 2026-06-02 18:00

## Overview

Implements the full `gwen benchmark` command across seven new files. All
measurement logic (cold-start process spawning, Ollama token throughput, in-
process dequantisation timing, and process memory sampling) lives in
`gwen-core`; `gwen-tui` calls only typed functions and owns progress printing
and report rendering. No new crate dependencies — uses only `std::time::Instant`,
`std::process::Command`, and the existing `reqwest::blocking` and `sysinfo`
deps already present in `gwen-core`. Zero new errors, zero new warnings
introduced. One edition-2024 keyword collision (`gen`) caught and fixed during
first `cargo check`.

## New / Modified Files

| File | Change |
|---|---|
| `packages/core/src/benchmark/mod.rs` | **New file.** Module registry plus orchestration. Declares `ColdStartResult`, `InferenceResult`, `ConvertBenchResult`, `MemoryResult`, `BenchmarkResult` (top-level aggregate with `Option` fields so individual suites can be skipped). `BenchmarkFilter` struct with `is_all()`, `run_*()` helpers converts CLI flag booleans into suite-selection logic. `ProgressCallback` type alias for the `[i/N] Suite ... ✓` live output. `run_benchmarks(filter, progress)` orchestrator calls each suite in order, tracks suite index dynamically so `[i/N]` is correct even when a subset of suites is selected, and measures total elapsed via `Instant`. Documents why criterion is not used (compile overhead, iteration-count overkill for target verification vs optimisation work). |
| `packages/core/src/benchmark/cold_start.rs` | **New file.** Spawns `std::env::current_exe() --version` 10 times via `std::process::Command`, measures wall-clock from `spawn()` to first stdout byte (`read_exact(&mut [0u8; 1])`). Comments explain why `current_exe` is preferred over hard-coded "gwen" (CI path, Windows extension, alias resolution). Why `--version` (clap intercepts before tokio starts → measures loader cost only). Why first-byte rather than exit (user-perceived latency). Why 10 iterations (stable median, < 150 ms total). `aggregate()` computes min/max/mean/median with sort; median uses average-of-two-middle for even N. |
| `packages/core/src/benchmark/inference.rs` | **New file.** Single blocking POST to `http://127.0.0.1:11434/api/generate` (`stream: false`) using `reqwest::blocking::Client` with 180 s timeout. Fixed prompt: `"Explain the concept of recursion in one paragraph."`. Checks Ollama reachability first with a 2 s GET to root. Returns `Option<InferenceResult>` — `None` when Ollama is unreachable. Uses 4-chars/token heuristic (same as eval/metrics.rs). Comments explain why blocking not async (orchestrator runs on plain thread, nested tokio would be unnecessary complexity), why `stream: false` (total roundtrip latency, not per-token), why fixed prompt (reproducibility). `gen` keyword collision avoided: variable named `generated` (edition 2024 reserves `gen` for generator syntax). |
| `packages/core/src/benchmark/convert_bench.rs` | **New file.** Builds a deterministic synthetic Q8_0 tensor (10 blocks × 32 elements, scale = 0.5 f16, sawtooth i8 values -16..+15 with per-block offset) and calls `crate::convert::dequant::dequantize(tensor, mode)` 1,000 times for each mode. Records per-call nanoseconds via `Instant`. Computes mean and population stddev of ns/element. Comments explain why synthetic (no file dependency), why Q8_0 (isolates arithmetic not nibble-unpacking), why 1,000 iterations (stable stddev, < 200 ms), why population stddev (Bessel correction negligible at N=1000, standard benchmark convention). Sawtooth values ensure Euler's `max_bound` is always non-zero, giving a fair comparison. |
| `packages/core/src/benchmark/memory.rs` | **New file.** `sample_memory() → MemoryResult`. Duplicates the `sample_peak_memory_mb` implementation from `eval/metrics.rs` (Linux: VmRSS from `/proc/self/status`; non-Linux: sysinfo `refresh_processes()` + `process(pid).memory()`). Not shared via a pub function in eval because `sample_peak_memory_mb` is private there; making it pub just for benchmark would couple two unrelated modules. Comment explains the duplication decision. |
| `packages/tui/src/commands/benchmark.rs` | **New file.** `BenchmarkArgs` (clap `Args`): `--cold-start`, `--inference`, `--convert`, `--full`. `run_benchmark_cmd(args)` builds `BenchmarkFilter`, prints `⏱ Running benchmarks...`, allocates a `ProgressCallback` that prints `[i/N] Suite ...` without newline (flushes stdout), then appends ` ✓` on the done call. `print_report` renders the spec-prescribed output: Gwen Orange section headers, per-suite rows, cold-start target-check hint (> 10 ms warning), Euler overhead ratio, Ollama skip warning. `should_show_inference_skip` infers whether inference was requested based on what else ran. |
| `packages/core/src/lib.rs` | **Modified:** Added `pub mod benchmark;` before `pub mod convert`. |
| `packages/tui/src/commands/mod.rs` | **Modified:** Added `pub mod benchmark;` (alphabetical order). |
| `packages/tui/src/main.rs` | **Modified:** Replaced `Benchmark` unit variant with `Benchmark(commands::benchmark::BenchmarkArgs)`. Replaced stub match arm with `Commands::Benchmark(args) => { commands::benchmark::run_benchmark_cmd(args); std::process::exit(0); }`. |

## Architecture Notes

| Item | Detail |
|---|---|
| Crate boundary | All timing, process spawning, HTTP, and memory measurement live in `gwen-core`. `gwen-tui`'s benchmark.rs calls only `run_benchmarks`, `BenchmarkFilter`, `BenchmarkResult`, and `ProgressCallback` — it never touches `std::process::Command`, `reqwest`, or `sysinfo` directly. |
| No new dependencies | `reqwest::blocking` and `sysinfo` are already in `gwen-core`'s Cargo.toml. `std::process::Command` and `std::time::Instant` are stdlib. No `criterion`, no `half`, no new entries in any Cargo.toml. |
| `gen` keyword (edition 2024) | `gen` is reserved in Rust edition 2024 for generator/coroutine syntax. The variable in `inference.rs` was renamed to `generated` (same fix pattern as `output_eval.rs` in the eval session). Comment in the file documents the reason. |
| Why `reqwest::blocking` in benchmark | The `run_benchmarks` orchestrator runs synchronously on the calling thread (no tokio context). Using `reqwest::async_std` would require `tokio::runtime::Builder::new_current_thread().block_on(...)` inside gwen-core — unnecessary complexity for a single HTTP call. The blocking client is already a declared feature of `reqwest` in `gwen-core`'s Cargo.toml. |
| Memory duplication rationale | `sample_peak_memory_mb` in `eval/metrics.rs` is a private `fn`. Promoting it to `pub` would expose an internal eval helper across module boundaries for a benchmark that is architecturally unrelated to eval. The 20-line duplication is the lower-coupling choice. |
