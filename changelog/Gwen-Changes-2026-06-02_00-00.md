# GwenCLI Change Log — 2026-06-02 00:00

## Overview

Implements the full `gwen eval` command across six new files. The feature spans both crates: all metric computation and data logic lives in `gwen-core`; `gwen-tui` calls typed functions and never touches sysinfo, reqwest, or serde internals directly. The command runs a two-phase evaluation pipeline with a live Ratatui TUI panel during execution and a colour-coded terminal summary afterwards. Zero new errors, zero new warnings introduced.

## New / Modified Files

| File | Change |
|---|---|
| `packages/core/src/eval/mod.rs` | **New file.** Module registry for the three eval sub-modules (`metrics`, `output_eval`, `report`). Documents the architectural split: metrics is sync/pure-Rust, output_eval is async/network-bound, report owns the JSON schema. |
| `packages/core/src/eval/metrics.rs` | **New file.** `EvalSample { input, output }` (JSONL record), `MetricsResult { avg_loss, perplexity, tokens_per_sec, memory_per_token_mb }`, `ProgressCallback`, `load_samples(path) → Result<Vec<EvalSample>>`, `compute_metrics(samples, callback) → Result<MetricsResult>`. Loss is a surrogate cross-entropy based on character-level bigram overlap (Ollama's `/api/generate` does not expose log-probs). Memory measurement is platform-split: Linux reads `VmRSS` from `/proc/self/status`; all other platforms use `sysinfo 0.30`'s `refresh_processes()` + `process(pid).memory()`. Returns `0.0` on any measurement failure so a missing `/proc` or sysinfo error never aborts eval. Token count estimated at 4 chars/token (same heuristic used in `dry_run.rs` and `chat.rs`). |
| `packages/core/src/eval/output_eval.rs` | **New file.** `SampleResult { input, expected, generated, match }`, `OutputProgressCallback`, `run_output_eval(samples, model, max_samples, callback) → Result<Vec<SampleResult>>`, `exact_match_rate(results) → f64`. Calls Ollama `/api/generate` (non-streaming, 180 s timeout) directly on port 11434 — bypasses the SSE proxy so `gwen serve` is not required. Match criterion: `generated.trim().to_lowercase().contains(expected.trim().to_lowercase())`. Inference errors are recorded as the `generated` field (non-fatal) so a single unresponsive model call does not abort the entire run. |
| `packages/core/src/eval/report.rs` | **New file.** `EvalReport { model, dataset, metrics: ReportMetrics, samples }`, `ReportMetrics { avg_loss, perplexity, tokens_per_sec, memory_per_token_mb, exact_match_rate }`, `ReportMetrics::from_parts(m, exact_match_rate)`, `write_report(report, path) → Result<()>`. Serialises to pretty-printed JSON (two-space indent) for human readability. |
| `packages/tui/src/commands/eval.rs` | **New file.** `EvalArgs` (clap `Args`: `--model`, `--dataset`, `--output`, `--max-samples`), `EvalPanelState` (shared Mutex state for TUI renderer and background worker), `render_eval_panel` (Ratatui panel: title bar, Phase 1 metrics, Phase 2 progress, Gauge with ETA, keybind hints), `run_eval_cmd(args, mode)`, `run_eval_inner(args)`, `print_final_report(report)`. Background worker runs both phases sequentially on a `std::thread` — Phase 1 synchronously, Phase 2 via a dedicated `tokio::runtime::Builder::new_current_thread()` (the outer tokio runtime cannot be entered from a `std::thread`). TUI event loop runs on the main thread with 100 ms poll; [Q] detaches cleanly. |
| `packages/core/src/lib.rs` | Added `pub mod eval;` between `dataset` and `train`. |
| `packages/tui/src/commands/mod.rs` | Added `pub mod eval;`. |
| `packages/tui/src/main.rs` | Replaced `Eval` stub variant with `Eval(commands::eval::EvalArgs)`. Removed `Eval` from the "coming soon" match arm. Added `Commands::Eval(args) => { commands::eval::run_eval_cmd(args, mode).await; std::process::exit(0); }`. |

## Architecture Notes

| Item | Detail |
|---|---|
| Crate boundary rule | `sysinfo`, `reqwest`, `serde_json` are deps of `gwen-core`. `gwen-tui`'s eval.rs calls only typed functions (`metrics::compute_metrics`, `output_eval::run_output_eval`, `report::write_report`) and never names those crates directly. |
| Surrogate loss | Ollama's `/api/generate` API returns generated text, not per-token log-probabilities. True cross-entropy requires logit access. The surrogate (bigram overlap → `-log(ratio)`) produces a monotone quality signal in the expected 1–150 PPL range without a logit API. The comment in `metrics.rs` documents the design decision and its limitation explicitly. |
| Platform memory measurement | Linux reads `VmRSS` from `/proc/self/status` (updated per allocation, no external crate). Non-Linux uses `sysinfo 0.30`'s `refresh_processes()` (no `ProcessesToUpdate` API in this version — full refresh is acceptable because memory is sampled only twice per eval run). Fallback to `0.0` is safe: `memory_per_token_mb = 0.0` is serialised to JSON and printed in the report without causing a panic. |
| Background thread + dedicated tokio runtime | Phase 2's async Ollama calls run inside `tokio::runtime::Builder::new_current_thread().block_on(...)` inside the `std::thread`. Calling `tokio::spawn` from a `std::thread` that is not inside the outer runtime would panic. The dedicated runtime is a one-liner that avoids cross-runtime complexity. |
| TUI event loop blocking model | `crossterm::event::poll` is synchronous. Running it from inside the outer `tokio` async context (e.g. via `tokio::task::spawn_blocking`) would require passing `Arc<Mutex<terminal>>` across thread boundaries, which crossterm does not support. The main thread owns the terminal exclusively; the background worker owns no terminal state. |
| `gen` keyword collision | `gen` is a reserved keyword in Rust edition 2024. The variable was renamed to `generated_lc` in `output_eval.rs`. |
