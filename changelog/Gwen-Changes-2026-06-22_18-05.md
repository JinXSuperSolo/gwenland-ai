## Executive Summary

This session turned the existing Tauri GUI shell into the standalone GwenLand Core desktop dashboard
described in the attached spec. The app now opens to a dark, warm-orange local toolkit instead of the
older proxy/chat-oriented surface.

The new GUI is organized around the operator workflows GwenLand needs most:

1. **Dashboard** - local runtime stats, quick actions, recent activity, and active training status.
2. **Inference** - model selector, prompt workspace, decode controls, output, and generation metrics.
3. **Train** - LoRA training setup with dataset/output pickers and live job status.
4. **Benchmark** - multi-model benchmark selection with throughput, latency, and memory results.
5. **Models** - local GGUF/GGQR registry, import, download, load, benchmark, and copy-path actions.
6. **Settings** - local binary path, model directory, default runtime knobs, and build notes.

Everything stays local. There is no cloud service, auth flow, account concept, or light-mode variant.

---

## Important Implementation Note

The attached spec asked for a Svelte + Phosphor frontend. The checked-in GUI package is already a
React + Vite Tauri app, and the workspace did not have Svelte or Phosphor packages installed. Rather
than introduce a dependency conversion under restricted network conditions, the implementation kept
the existing React/Vite shell and built the requested product surface there.

That kept the app buildable with the packages already in the repo. The UI still follows the spec's
page structure, local-only behavior, Tauri command boundary, warm dark/orange visual language, and
desktop-dashboard intent.

## What Changed

### 1. Tauri command surface for Core workflows

`packages/gui/src-tauri/src/lib.rs` now exposes the GUI-facing command layer:

- `run_inference`
- `start_training`
- `get_train_status`
- `run_benchmark`
- `list_models`
- `import_model`
- `download_model`
- `get_system_stats`

The command layer resolves the GwenLand CLI from `GWENLAND_BIN`, sibling installed binaries, the
Tauri sidecar directory, or finally `gwenland(.exe)` on `PATH`. It shells out through the local CLI,
parses JSON envelopes when present, and falls back to readable output when needed.

Training jobs are tracked in memory and write stdout/stderr logs under the GwenLand local jobs
directory. The status command polls the child process, extracts step/loss progress from recent logs,
and returns a compact status object for the frontend monitor.

Model management reads from the GwenLand models directory, accepts `.gguf` and `.ggqr`, infers common
quantization/parameter tags from filenames, and returns stable metadata for the UI.

### 2. New Core dashboard frontend

`packages/gui/src/App.tsx` was rebuilt around the six spec pages. The old provider-heavy proxy/chat
shell was removed from the app entry path, and `main.tsx` now mounts the dashboard directly inside the
existing error boundary.

The UI includes:

- a fixed Core sidebar with Dashboard, Inference, Train, Benchmark, Models, and Settings;
- top-level active model selector;
- runtime stat cards for memory, cold start, binary size, and model count;
- prompt presets and decode sliders;
- training job progress, log tail, and loss bars;
- benchmark comparison rows with throughput bars;
- model import/download controls and per-model actions.

The app handles empty states cleanly, especially the common first-run state where no local models are
registered yet.

### 3. Warm Core visual system

`packages/gui/src/index.css` now defines the GUI's dark desktop theme:

- base: `#1b1918`
- surface/sidebar: warm black and brown-gray layers
- primary: `#b56936`
- compact cards with 8px radius
- self-hosted font aliases mapped onto the existing bundled Geist files
- responsive layouts for narrow desktop and mobile-sized webview testing

The first visual pass found one real layout issue: the dashboard hero copy was squeezed into a narrow
column beside the quick-action buttons. The panel was adjusted so the copy spans naturally and the
actions sit below it.

### 4. Tauri app identity

`tauri.conf.json` now presents the window and product as **GwenLand Core**, with a slightly larger
default/minimum desktop size:

```text
width:     1280
height:    820
minWidth:  960
minHeight: 640
```

The package description was also updated to match the new dashboard role.

## Deliberate Boundaries

This work did not convert the GUI package from React/Vite to Svelte/Vite. That can still be done later
as a separate dependency and migration task, but doing it here would have made the dashboard depend on
new packages that were not present in the workspace.

The backend command layer is intentionally thin. It calls the local GwenLand CLI/subprocess path
instead of duplicating inference, training, benchmark, or fetch logic inside the Tauri crate.

The GUI does not add authentication, accounts, remote sync, cloud model calls, or a light theme.
