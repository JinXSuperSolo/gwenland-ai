# GwenLand — GGQR-CF-mmap Dequantisation Engine: Q4_K, Q5_K, Q6_K

**Date:** 2026-06-07
**Scope:** `gwen-cli/packages/core/src/convert/`
**Type:** Feature — dequantisation engine extension
**Status:** ⚠️ EXPERIMENTAL — not validated against real GGUF model files end-to-end.
Bit manipulation verified against GGML reference source. Test helpers only
round-trip correctly for sub-block scale/min values < 16. Cross-check output
against a known-good dequantiser (e.g. llama.cpp) before promoting to stable.

---

## Summary

Extends the GGQR-CF-mmap dequantisation engine with three K-quant formats:
**Q4_K**, **Q5_K**, and **Q6_K**. These are the dominant quantisation formats
used by modern GGUF models (Qwen3, LLaMA 3.1, Mistral, Gemma 2) and were
previously unsupported, causing `gwen convert` to fail with an "unsupported
ggml_type" error on any real-world model file. This change makes `gwen convert`
production-ready for the full GGUF ecosystem.

---

## Why

### The Problem

The existing dequantisation engine (`dequant.rs`) supported only four dtypes:
`F32`, `F16`, `Q8_0`, and `Q4_0`. In practice, virtually every quantised GGUF
model distributed today uses K-quant formats — `Q4_K_M`, `Q5_K_M`, `Q6_K` —
because they achieve significantly better quality-per-bit than the legacy
`Q4_0`/`Q8_0` formats. Running `gwen convert` on any Qwen3, LLaMA 3.1, or
Mistral GGUF file would immediately fail with:

```
tensor 'token_embd.weight' has unsupported ggml_type 12
(only F32/F16/Q4_0/Q8_0 are supported)
```

This blocked the entire `gwen convert` pipeline for real-world usage.

### Why These Three Formats

| Format | GGML type ID | Bits/weight | Common use |
|--------|-------------|-------------|------------|
| Q4_K   | 12          | 4.5         | Q4_K_M, Q4_K_S — most popular quantisation |
| Q5_K   | 13          | 5.5         | Q5_K_M — best quality/size tradeoff |
| Q6_K   | 14          | 6.5625      | Q6_K — near-lossless, large models |

These three cover the overwhelming majority of GGUF files in the wild. Adding
them makes `gwen convert` functional for essentially all quantised models.

### Why Now

Cycle 6 shipped native inference (`gwen run`) which loads GGUF files directly.
The convert pipeline (`gwen convert`) is the complementary path for users who
want to export to SafeTensors for HuggingFace tooling or GwenTensor inference.
Without K-quant support, convert was a dead feature for any model built after
2024.

## What

### New Functions — `dequant.rs`

Six new dequantisation functions following the exact same pattern as the
existing `dequant_q8_0_standard` / `dequant_q8_0_euler` pair:

| Function | Mode | Description |
|---|---|---|
| `dequant_q4_k_standard` | Standard | Linear: `W = d·scale[j]·q − dmin·min[j]`, q∈[0,15] |
| `dequant_q4_k_euler` | Euler | Cosine projection via `euler_dequant_block` |
| `dequant_q6_k_standard` | Standard | Linear: `W = d·scales[j]·q`, q∈[-32,31] |
| `dequant_q6_k_euler` | Euler | Cosine projection via `euler_dequant_block` |
| `dequant_q5_k_standard` | Standard | Linear: `W = d·scale[j]·q − dmin·min[j]`, q∈[0,31] |
| `dequant_q5_k_euler` | Euler | Cosine projection via `euler_dequant_block` |

### Superblock Layouts (GGML-verified)

**Q4_K — 144 bytes / 256 elements:**
```
[d: f16][dmin: f16][scales: u8×12][nibbles: u8×128]
```
- 8 sub-blocks × 32 elements
- 6-bit scale + 6-bit min per sub-block (packed via GGML `get_scale_min_k4`)
- 4-bit unsigned values, q∈[0,15]

**Q6_K — 210 bytes / 256 elements:**
```
[ql: u8×128][qh: u8×64][scales: i8×16][d: f16]
```
- 16 sub-blocks × 16 elements
- 6-bit values reconstructed from 4-bit low + 2-bit high parts
- Signed output q∈[-32,31]

**Q5_K — 176 bytes / 256 elements:**
```
[d: f16][dmin: f16][scales: u8×12][qh: u8×32][ql: u8×128]
```
- 8 sub-blocks × 32 elements (same scale/min packing as Q4_K)
- 5-bit values = 4-bit low (from ql) + 1 high bit (from qh, 8 per byte)
- Unsigned output q∈[0,31]

### Updated Dispatch — `dequantize()`

```rust
GgufDtype::Q4_K => match mode {
    DequantMode::Standard => dequant_q4_k_standard(&tensor.raw_data, n_elements),
    DequantMode::Euler    => dequant_q4_k_euler(&tensor.raw_data, n_elements),
},
GgufDtype::Q5_K => match mode {
    DequantMode::Standard => dequant_q5_k_standard(&tensor.raw_data, n_elements),
    DequantMode::Euler    => dequant_q5_k_euler(&tensor.raw_data, n_elements),
},
GgufDtype::Q6_K => match mode {
    DequantMode::Standard => dequant_q6_k_standard(&tensor.raw_data, n_elements),
    DequantMode::Euler    => dequant_q6_k_euler(&tensor.raw_data, n_elements),
},
```

### Updated Parser — `gguf_parser.rs`

Three new `GgufDtype` variants added to the enum:

```rust
Q4_K,  // ggml_type 12 — 144 bytes/superblock
Q5_K,  // ggml_type 13 — 176 bytes/superblock
Q6_K,  // ggml_type 14 — 210 bytes/superblock
```

`ggml_type_to_dtype()` now maps IDs 12, 13, 14 to the new variants.
`raw_size_bytes()` computes the correct on-disk byte count for each.

### New Documentation — `math_theory_q4k.md`

Mirrors the style of the existing math theory documentation. Covers:
- Background: K-quants vs. legacy quants (why the two-level hierarchy)
- Q4_K: superblock layout, `get_scale_min_k4` packing derivation, formula
- Q6_K: bit reconstruction (4-bit low + 2-bit high), signed output range
- Q5_K: 5th-bit extraction (8 bits per byte in qh), formula
- Comparison table (bits/weight, sub-block count, formula, q range)
- Euler mode interaction (why min is not applied in Euler mode)

## Questions & Answers

**Q: Why is Q6_K implemented before Q5_K in the file, even though Q5_K has higher priority?**

A: Q6_K is structurally simpler — it has a single scale per superblock (no `dmin`
term) and a straightforward 4+2 bit reconstruction. Implementing it second (after
Q4_K) and before Q5_K lets the reader build up complexity incrementally: Q4_K
introduces the superblock + sub-block scale/min concept, Q6_K introduces the
multi-byte bit reconstruction, and Q5_K combines both. The dispatch order in
`dequantize()` follows the GGML type ID order (12, 13, 14) regardless.

**Q: Why is the `dmin` (min) term not applied in Euler mode?**

A: The Euler cosine projection (`W = cos(θ) × δ_b / φ`) already bounds the output
to `[-0.618, 0.618]`. Subtracting a per-sub-block `dmin` offset after the
projection would break this bound and introduce asymmetry that the GwenTensor
fixed-point accumulator does not expect. The `d` superblock scale is preserved as
the amplitude modulator `δ_b`, which carries the per-block magnitude information.
This is the same design decision made for Q4_0 Euler mode.

**Q: Why use a stack array `[u8; 8]` for `sub_scales`/`sub_mins` instead of a Vec?**

A: The hard requirement explicitly prohibits per-block heap allocation. The
sub-block count is a compile-time constant (8 for Q4_K and Q5_K, 16 for Q6_K).
Stack arrays are zero-cost — no allocation, no drop, no cache miss. The Q6_K
Standard path avoids the array entirely by computing the sub-block index inline
(`j = i / SUBBLOCK_ELEMENTS`).

**Q: How was the bit manipulation verified against GGML?**

A: Each bit reconstruction was cross-checked against the GGML reference
implementations:
- Q4_K: `dequantize_row_q4_K` + `get_scale_min_k4` in `ggml-quants.c`
- Q6_K: `dequantize_row_q6_K` in `ggml-quants.c`
- Q5_K: `dequantize_row_q5_K` in `ggml-quants.c`

The unit tests verify the boundary values (q=0 → min output, q=max → max output)
which are the most likely places for off-by-one errors in bit manipulation.

**Q: Will this break existing `gwen convert` calls on Q8_0/Q4_0 models?**

A: No. The existing functions are completely untouched. The dispatch `match` in
`dequantize()` adds new arms without modifying existing ones. The `GgufDtype`
enum is extended with new variants; Rust's exhaustive match ensures any future
dtype addition will produce a compile error rather than a silent fallthrough.

**Q: What happens if a GGUF file contains a dtype not yet supported (e.g. Q2_K)?**

A: `ggml_type_to_dtype()` returns an `Err(String)` with a descriptive message
listing all supported types. This propagates up through `parse()` →
`convert_gguf()` → the TUI layer, which prints it as a clean error. No panic,
no silent corruption.

**Q: Are there plans for Q2_K, Q3_K, IQ formats?**

A: Not in this cycle. Q4_K, Q5_K, and Q6_K cover the formats used by the models
GwenLand targets (Qwen3, LLaMA 3.1, Mistral, Gemma 2). Q2_K and Q3_K are
lower-quality formats rarely used for inference. IQ (importance-matrix) formats
require additional metadata not yet parsed by the GGUF parser. These can be
added in a future cycle by following the same pattern established here.
