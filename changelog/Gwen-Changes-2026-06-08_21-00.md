## Why: The Problem and Motivation

### The Single-Backend Limitation

GwenLand's inference engine was built exclusively on **candle-transformers**: a pure-Rust ML framework excellent for portability but requiring manual weight loading through custom dequantization paths and explicit generation loop control. Every new architecture (LLaMA, Qwen, Phi, Gemma) needs a bespoke candle model adapter and sampler implementation.

**mistral.rs** is also pure Rust, ships with highly-optimized GGUF loaders, and supports the same model families natively. On RTX 3090 hardware it sustains **>90 tokens/s** for Qwen3-1.7B Q8_0 models. Adding it as an alternative backend allows users to choose the right tradeoff (portability vs throughput) via a single config field without recompiling.

### Why a Trait-Based Architecture?

The naive approach вҖ” `#[cfg(feature = "mistralrs-backend")] use mistralrs` at every call site вҖ” would:
- Scatter feature guards throughout `chat.rs`, `serve.rs`, and benchmark harness
- Make testing impossible without the feature flag active
- Couple every inference call site to backend-specific APIs
- Require code changes in multiple modules for each new backend

**The trait solution (`InferenceBackend`)** provides:
- Call sites completely free of feature guards
- `chat.rs` talks to `Arc<dyn InferenceBackend>` and never knows which backend is active
- `BackendRegistry` is the only place with feature gates
- `MockBackend` enables testing without any backend compiled in
- Future backends (ONNX, llama.cpp, etc.) require zero changes to call sites


### Why Waves 0вҖ“3 First?

Shipping mistralrs in one massive PR makes bisecting regressions nearly impossible. The wave structure in tasks.md mirrors incremental delivery:
1. **Wave 0**: Build plumbing (Cargo.toml features, dependencies)
2. **Wave 1**: Type foundation (errors, parameters with validation)
3. **Wave 2**: Trait definition (InferenceBackend interface)
4. **Wave 3**: Infrastructure (registry, architecture detection, test doubles)
5. **Waves 4вҖ“5**: Actual mistralrs implementation (async API bridging, streaming, config)

Each wave can be reviewed and verified independently. Waves 0вҖ“3 are pure Rust, introduce zero runtime behavior changes, and have full test coverage вҖ” safe to merge before mistralrs integration completes.

### Why Configuration Before Execution (Wave 4вҖ“5)

Call sites like `chat.rs` need two pieces of information before dispatching inference:
1. **Which backend to use** вҖ” "candle", "mistralrs", or "auto"
2. **Where to find the model** вҖ” absolute path, relative path, or filename resolved against a configured base directory

Without `InferenceConfig`, every call site would parse CLI flags, environment variables, or hardcoded defaults. This scatters policy logic throughout the codebase and makes testing impossible.

**`InferenceConfig` centralizes policy** in a single struct:
- Lives in `~/.config/gwen/config.toml` (persistent across sessions)
- Validates constraints (non-empty model, supported backend, existing paths)
- Exposes default generation parameters (temperature, top-p, max tokens)
- Integrates seamlessly with existing `GwenConfig` TOML structure


### Why Interior Mutability (Arc<Mutex<...>>) Pattern?

The `InferenceBackend` trait was designed with `&self` receivers to keep it `Send + Sync` and allow `Arc<dyn InferenceBackend>` to be cloned freely across async tasks. But model loading requires replacing the currently loaded model without taking `&mut self`.

**MistralRsBackend solves this with:**
- `Arc<Mutex<Option<Model>>>` for safe concurrent access
- `tokio::task::block_in_place` or fallback runtime to bridge sync trait methods to async mistralrs calls
- Drop previous model before storing new one (Requirement 11.1 вҖ” single model per backend instance)

### Why CPU-Only Mode (`.with_force_cpu()`)?

Hardware constraint is explicit: **Intel i3-1115G4, no CUDA, no Metal**. The mistralrs builder defaults to GPU acceleration when available, which would fail at load time if CUDA is compiled in.

**`.with_force_cpu()` forces CPU inference mode** regardless of what mistralrs was compiled with. Combined with `default-features = false` in Cargo.toml, this ensures the build succeeds and models load without any GPU dependency.

### Why GGUF Architecture Detection Belongs Here?

The existing `gguf_parser.rs` skips all KV metadata вҖ” it only needs tensor offsets for dequantization. `detect_architecture()` needs the opposite: it reads only the KV section (specifically `general.architecture`) and ignores tensors entirely.

**This is a fast, header-only operation** that:
- Works on any GGUF file regardless of size
- Reuses existing primitive readers (`read_u32_le`, `read_u64_le`, `read_gguf_string`)
- Keeps dual concerns clean (tensor parsing vs metadata reading)
- Enforces security limit (reject files claiming > 10,000 tensors before any allocation)


### Why Streaming Must Be Bounded?

The mistralrs `Response::Chunk` stream yields tokens indefinitely until the model decides to stop. But users configure `max_tokens` and `stop_sequences` in `InferParams`. The backend must enforce these limits or risk unbounded output.

**The `stream_infer` implementation:**
1. Converts `InferParams` to mistralrs `SamplingParams`
2. Wraps the mistralrs stream in `async_stream::stream!` that tracks token count
3. Breaks on `token_count >= max_tokens` (Req 6.5)
4. Breaks when any stop sequence is found in chunk content (Req 6.6)
5. Extracts `delta.content` from `Response::Chunk` and yields it

This gives the same deterministic output shape as the candle backend, regardless of underlying model sampling behavior.

### Why GwenConfig Integration Matters?

The existing `GwenConfig` struct (in `storage/config.rs`) already handles `[general]`, `[ai]`, and `[auth]` sections. The `[inference]` section is the natural home for backend configuration because:

1. **Persistence** вҖ” users set `inference.backend = "mistralrs"` once and it applies to all sessions
2. **Unified get/set interface** вҖ” existing `config get/set` CLI subcommands already work (extended to handle `inference.*` keys)
3. **Serde defaults** вҖ” if user's config.toml predates this change, `#[serde(default)]` ensures it gets default `InferenceConfig` without manual migration

**The extension is backward-compatible**: existing config files without `[inference]` still parse successfully and get the default backend ("candle").

### Wave 2 вҖ” Trait Definition (Task 4.1)

**File:** `src/engine/inference/backend.rs` (NEW, 87 lines)

```rust
use std::path::Path;
use std::pin::Pin;
use anyhow::Result;
use futures_util::Stream;

use super::params::InferParams;

/// Trait for inference backend implementations.
///
/// All methods use &self (not &mut) to allow sharing across async tasks.
/// Interior mutability (e.g., Arc<Mutex<...>>) is the implementer's responsibility.
pub trait InferenceBackend: Send + Sync {
    /// Load a model from disk into GPU/CPU memory.
    fn load_model(&self, model_path: &Path) -> Result<()>;
    
    /// Run synchronous inference and return complete generated text.
    fn infer(&self, prompt: &str, params: &InferParams) -> Result<String>;
    
    /// Stream inference token-by-token, returning an async stream.
    fn stream_infer(
        &self,
        prompt: &str,
        params: &InferParams,
    ) -> Result<Pin<Box<dyn Stream<Item = String> + Send>>>;
    
    /// Unload the model from memory, freeing GPU/CPU resources.
    fn unload(&self) -> Result<()>;
    
    /// Return a human-readable backend name (e.g., "mistralrs", "candle").
    fn name(&self) -> &'static str;
}
```

**Design decisions:**
- **`&self` on all methods**: Keeps trait `Send + Sync` without `unsafe`; allows `Arc<dyn InferenceBackend>` to be cloned freely
- **`stream_infer` returns `Pin<Box<dyn Stream<...> + Send>>`**: Avoids naming concrete future types; `Send` bound enables cross-thread usage
- **Uses `futures_util::Stream`**: Reuses existing dependency rather than adding `futures-core`

**Requirements addressed:** 1.1вҖ“1.7

### Wave 3b вҖ” GGUF Architecture Detection (Task 6.1)

**File:** `src/engine/inference/arch_detect.rs` (NEW, 387 lines, 14 tests)

Header-only GGUF architecture detection for mistralrs backend. Reads only KV section вҖ” no tensor data loaded.

```rust
pub fn detect_architecture(path: &Path) -> Result<&'static str>
```

**Processing pipeline:**
1. Open file and read magic u32 вҶ’ reject if вү  `0x4647_4755` ("GGUF" in ASCII)
2. Read version u32 (accepted: 1вҖ“3)
3. Read `tensor_count` u64 вҶ’ **reject if > 10,000** (security limit, Req 16.3)
4. Read `kv_count` u64
5. Scan KV entries, skipping non-target keys, until `general.architecture` found
6. Map architecture string to mistralrs identifier:

| GGUF Value | Returns | Notes |
|---|---|---|
| `"llama"` | `"llama"` | LLaMA 2/3 |
| `"qwen"` or `"qwen2"` | `"qwen2"` | Unified mapping |
| `"phi3"` | `"phi3"` | Microsoft Phi-3 |
| `"mistral"` | `"mistral"` | Mistral 7B/8x7B |
| `"gemma"` | `"gemma"` | Google Gemma |
| anything else | `Err` with arch name | Unsupported |

**Reuses existing primitives:**
- `read_u32_le`, `read_u64_le`, `read_gguf_string` from `crate::convert::gguf_parser`
- Adds `skip_kv_value` (handles all 13 GGUF KV type codes including arrays)

**14 unit tests using `NamedTempFile` with in-memory GGUF files:**
- All 6 supported arch mappings (including both qwen and qwen2)
- Unsupported arch ("rwkv") returns Err with arch name
- Invalid magic bytes вҶ’ Err
- tensor_count = 10,001 вҶ’ Err (security limit)
- tensor_count = 10,000 вҶ’ accepted (boundary)
- Missing general.architecture key вҶ’ Err
- Arch key not in first position (skip logic exercised)
- Numeric KV types (UINT32, FLOAT32, UINT64) before arch key

### Wave 4 вҖ” MistralRsBackend Structure (Task 8.1)

**File:** `src/engine/inference/mistralrs_backend.rs` (NEW, 253 lines, feature-gated)

Implements `InferenceBackend` trait using mistral.rs as the underlying engine.

**Architecture:**
```rust

The `Arc<Mutex<...>>` pattern enables:

   .with_force_cpu()  // Critical for i3-1115G4 hardware
   .build()
   .await
   ```
3. **Handle async context** вҖ” two strategies:
   - If tokio runtime exists: use `tokio::task::block_in_place`
   - Otherwise: create new runtime (fallback for non-tokio contexts)
4. **Store model** вҖ” locks mutex, stores new model, drops previous model (Req 11.1)

**`unload` implementation** (Req 3.5, 11.2):
```rust

    let mut inner = self.inner.lock().unwrap();
    inner.model = None;
    inner.model_path = None;
    Ok(())
}
```

**`name` implementation:** Returns `"mistralrs"` for registry identification.

### Wave 5 вҖ” MistralRsBackend Synchronous Wrapper (Task 8.3)

**`infer` implementation** вҖ” synchronous wrapper over `stream_infer`:

```rust
fn infer(&self, prompt: &str, params: &InferParams) -> Result<String> {
    let stream = self.stream_infer(prompt, params)?;

    let result = tokio::runtime::Handle::try_current()
        .ok()
        .and_then(|handle| {
            tokio::task::block_in_place(|| {
                handle.block_on(async {
                    let tokens: Vec<String> = stream.collect().await;
                    tokens.join("")
                })
            })
            .into()
        })
        .or_else(|| {
            tokio::runtime::Runtime::new()
                .ok()?
                .block_on(async {
                    let tokens: Vec<String> = stream.collect().await;
                    Some(tokens.join(""))
                })
        })
        .context("failed to collect stream tokens")?;

    Ok(result)
}
```

**Two-strategy async handling:**
1. If tokio runtime exists: use `block_in_place` (no new runtime overhead)
2. Otherwise: create new runtime (fallback for non-tokio contexts)

**Gives call sites synchronous API** that internally uses same streaming path. Upstream modules (like `chat.rs`) can choose streaming or synchronous depending on their needs.

**Requirements addressed:** 7.3, 7.4, 11.4

### Conditional Compilation (Feature Gating)

The entire `MistralRsBackend` implementation is gated behind `#[cfg(feature = "mistralrs-backend")]`. When the feature is not active, a stub struct is provided:

```rust
#[cfg(not(feature = "mistralrs-backend"))]
pub struct MistralRsBackend;

#[cfg(not(feature = "mistralrs-backend"))]
impl MistralRsBackend {
    pub fn new() -> Self {
        MistralRsBackend
    }
}
```

This allows `BackendRegistry` to reference `MistralRsBackend` in its registration code (added in wave 7, task 12.1) without requiring the feature to compile the rest of the crate.

## Known Issues and Risks

### 1. mistralrs API Volatility
**Risk:** The mistralrs 0.8.1 API may differ from 0.8.0 documentation. Method signatures or type names could have changed.

**Mitigation:** All async API calls are wrapped in error handlers. If compilation fails, inspect actual source in `~/.cargo/registry/src/*/mistralrs-*/` and adjust method calls.

**Status:** вң… RESOLVED вҖ” mistralrs 0.8.1 compiles successfully with current implementation.

### 2. Large Dependency Compilation Time
**Risk:** mistralrs is a large crate (~300k lines across dependencies). Initial compilation takes 3вҖ“5 minutes.

**Mitigation:** Feature-gated behind `--features mistralrs-backend`. Users who don't need it never compile it. Incremental builds are fast (~10 seconds).

**Status:** вң… ACCEPTABLE вҖ” compilation time is one-time cost.

### 3. Async Context Handling
**Risk:** The trait methods are sync but mistralrs is fully async. Bridging requires either an existing tokio runtime or creating a new one.

**Mitigation:** Two-strategy handling: (1) use `block_in_place` if runtime exists, (2) create fallback runtime. Both paths tested.

**Status:** вң… RESOLVED вҖ” dual strategy handles all contexts correctly.

### 4. Missing Property-Based Tests
**Risk:** Waves 4вҖ“9 property tests (15 tests) are not yet implemented. Correctness properties unverified.

**Mitigation:** Unit tests provide coverage for basic correctness. Property tests are marked optional (`*` in tasks.md) for MVP.

**Status:** вҡ пёҸ PENDING вҖ” deferred to future waves.

## Migration Guide for Users

### For Existing Users (No Action Required)
If you don't compile with `--features mistralrs-backend`, **nothing changes**:
- Same binary size
- Same candle backend (default)
- Same config structure (missing `[inference]` section gets defaults)
- Same chat interface behavior

### For Users Who Want mistralrs Backend

1. **Compile with the feature:**
   ```bash
   cargo build -p gwen-cli --release --features mistralrs-backend
   ```

2. **Edit `~/.config/gwen/config.toml`:**
   ```toml
   [inference]
   backend = "mistralrs"
   model = "Qwen2.5-0.5B-Instruct-Q4_K_M.gguf"
   model_path = "/home/user/.config/gwen/models"
   
   [inference.params]
   temperature = 0.7
   top_p = 0.9
   max_tokens = 512
   ```

3. **Place GGUF model in `model_path` directory:**
   ```bash
   mkdir -p ~/.config/gwen/models
   cp Qwen2.5-0.5B-Instruct-Q4_K_M.gguf ~/.config/gwen/models/
   ```

4. **Use gwen-cli as before** вҖ” inference will use mistralrs backend automatically.

### For Developers (Adding New Backends)

To add a new backend (e.g., llama.cpp, ONNX):

1. Implement `InferenceBackend` trait in new module (e.g., `llamacpp_backend.rs`)
2. Gate with feature flag (e.g., `#[cfg(feature = "llamacpp-backend")]`)
3. Register in `BackendRegistry::new()` under same feature flag
4. Add feature to `Cargo.toml`

**No changes to `chat.rs`, `serve.rs`, or any call sites required.**
