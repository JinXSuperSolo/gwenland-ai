# GwenLand — GWEN-211: Fix MistralRsBackend for mistralrs 0.8.1 API

**Date:** 2026-06-09 12:00
**Scope:** `gwen-cli/packages/core/src/engine/inference/mistralrs_backend.rs`
**Type:** Bug Fix — align Wave 4–5 implementation to actual mistralrs 0.8.1 public API
**Status:** ✅ STABLE — `cargo check --features mistralrs-backend` 0 errors, 178 tests pass

---

## Executive Summary

The `MistralRsBackend` implementation written in Waves 4–5 was drafted against
documentation and assumed API shapes that turned out to differ from the actual
published `mistralrs 0.8.1` crate. When compiled with `--features mistralrs-backend`,
13 compiler errors blocked the feature-gated code path entirely.

This fix audits the real crate source at
`~/.cargo/registry/src/.../mistralrs-0.8.1/` and `mistralrs-core-0.8.1/`,
corrects every mismatched type and missing variant, and closes the gap. The
default build (no feature) and all 178 existing unit tests remain unaffected.

---

## Why

### Why the Original Code Was Wrong

The Wave 4–5 implementation was written by reading the mistralrs README and
changelog rather than the actual compiled crate source. The public API surface
of mistralrs 0.8.1 differs from both the 0.7 docs and the 0.8 prerelease
design notes in several specific ways. The errors were not caught earlier
because `cargo check` on the default build never compiles the
`#[cfg(feature = "mistralrs-backend")]` block — all 13 errors lived behind the
feature gate and were invisible to the standard CI path.

### Why Fix Now (Not Wave 9)

The original plan deferred this to Wave 9 ("API fix pending"). But Waves 6–8
have now wired `MistralRsBackend` directly into `BackendRegistry::new()` and
made it reachable from `run_headless_chat`. Leaving 13 compiler errors in a
code path that is now registered and callable is a liability: anyone who builds
with `--features mistralrs-backend` to test the native path gets an immediate
compile failure. Fixing it here unblocks that path and makes Wave 9 purely
about CLI flags and property tests rather than also carrying broken
infrastructure.

### Why Read the Crate Source Directly

The mistralrs changelog and GitHub README are not authoritative. The only
ground truth is the source in `~/.cargo/registry/src/`. Each of the nine bugs
below was diagnosed by reading:
- `mistralrs-0.8.1/src/lib.rs` — re-exports, what's actually public
- `mistralrs-0.8.1/src/messages.rs` — `TextMessages`, `RequestBuilder`, `TextMessageRole`
- `mistralrs-core-0.8.1/src/sampler.rs` — `SamplingParams` struct fields and `StopTokens`
- `mistralrs-core-0.8.1/src/response.rs` — `Response` enum variant definitions

## What: Nine Bugs Fixed

### Bug 1 — `mistralrs::Message` struct does not exist

**Before:**
```rust
let messages = mistralrs::TextMessages {
    messages: vec![mistralrs::Message {
        role: "user".to_string(),
        content: prompt.to_string(),
    }],
    sampling_params,
};
```

**After:**
```rust
let request = mistralrs::RequestBuilder::from(
    mistralrs::TextMessages::new()
        .add_message(mistralrs::TextMessageRole::User, prompt),
)
.set_sampling(sampling_params);
```

**Why:** `TextMessages` is a builder-only type — its `messages` field is private
(`Vec<IndexMap<String, MessageContent>>`). There is no public `Message` struct.
The correct API is `.new().add_message(role, text)`. The role is a
`TextMessageRole` enum (`User`, `Assistant`, `System`, `Tool`, `Custom(String)`),
not a `String`.

---

### Bug 2 — `TextMessages` struct literal used for non-public fields

Same root as Bug 1. `TextMessages { messages: ..., sampling_params: ... }` is a
struct literal for an opaque type. Struct literals require all fields to be
public and named exactly. Neither condition holds.

**Fix:** Already covered by switching to `RequestBuilder::from(TextMessages::new()...)`.

---

### Bug 3 — Custom `SamplingParams` silently ignored via `TextMessages`

**Before:** `SamplingParams` was constructed and assigned to the (nonexistent)
`TextMessages::sampling_params` field.

**Why this matters even after fixing Bug 1:** `TextMessages` implements
`RequestLike::take_sampling_params()` by always returning
`SamplingParams::deterministic()` (top_k=1, greedy). Custom temperature, top_p,
and stop sequences would have been silently discarded.

**After:** `RequestBuilder` is the only `RequestLike` implementor that accepts
a custom `SamplingParams`. `RequestBuilder::set_sampling(params)` replaces the
internal sampling config. Converting via `RequestBuilder::from(TextMessages)`
preserves the messages while allowing sampling override:

```rust
let request = mistralrs::RequestBuilder::from(text_messages)
    .set_sampling(sampling_params);
```

---

### Bug 4 — `SamplingParams::top_k` is `Option<usize>`, not `Option<i32>`

**Before:**
```rust
top_k: params.top_k.map(|k| k as i32),
```

**After:**
```rust
sp.top_k = params.top_k;  // already Option<usize>, no cast needed
```

**Why:** `SamplingParams.top_k` in `mistralrs-core-0.8.1/src/sampler.rs` is
typed as `Option<usize>`. Our `InferParams.top_k` is also `Option<usize>`.
The `as i32` cast was incorrect and would have truncated values > `i32::MAX`
(irrelevant in practice, but still wrong).

---

### Bug 5 — `SamplingParams` has no `Default` implementation

**Before:**
```rust
let sampling_params = mistralrs::SamplingParams {
    temperature: Some(...),
    top_k: ...,
    ..Default::default()   // ← compile error: Default not implemented
};
```

**After:**
```rust
let mut sp = mistralrs::SamplingParams::neutral();
sp.temperature = Some(params.temperature as f64);
sp.top_k = params.top_k;
sp.top_p = Some(params.top_p as f64);
// etc.
```

**Why:** `SamplingParams` derives `Clone, Debug, Serialize, Deserialize` but
not `Default`. Two named constructors exist: `neutral()` (all `None`, `n_choices=1`)
and `deterministic()` (forces `top_k=Some(1)` for greedy decoding). `neutral()`
is the right base for a custom sampling config because it makes no implicit
sampling decisions.

---

### Bug 6 — `stop_toks` expects `Option<StopTokens>`, not `Option<Vec<String>>`

**Before:**
```rust
stop_toks: if params.stop_sequences.is_empty() {
    None
} else {
    Some(params.stop_sequences.clone())  // Vec<String>, wrong type
},
```

**After:**
```rust
sp.stop_toks = if params.stop_sequences.is_empty() {
    None
} else {
    Some(mistralrs::StopTokens::Seqs(params.stop_sequences.clone()))
};
```

**Why:** `StopTokens` is an enum with two variants:
```rust
pub enum StopTokens {
    Seqs(Vec<String>),   // stop on any of these exact strings
    Ids(Vec<u32>),       // stop on any of these token IDs
}
```
We have strings, so `StopTokens::Seqs` is correct. Passing a bare `Vec<String>`
would not implement `Into<StopTokens>` — it is a type mismatch.

---

### Bug 7 — `Response::Done` is a tuple variant, not a unit variant

**Before:**
```rust
mistralrs::Response::Done => {
    break;
}
```

**After:**
```rust
mistralrs::Response::Done(_) => break,
```

**Why:** In `mistralrs-core-0.8.1/src/response.rs`:
```rust
pub enum Response {
    Done(ChatCompletionResponse),   // tuple variant — carries the final response
    Chunk(ChatCompletionChunkResponse),
    // ...
}
```
Matching a tuple variant without `(_)` or `(binding)` is a compile error.
We discard the payload because the stream is already exhausted at `Done`.

---

### Bug 8 — `Response::Error` variant does not exist

**Before:**
```rust
mistralrs::Response::Error(err) => {
    eprintln!("❌ mistralrs stream error: {}", err);
    break;
}
```

**After:**
```rust
mistralrs::Response::InternalError(e) => {
    eprintln!("mistralrs internal error: {e}");
    break;
}
mistralrs::Response::ValidationError(e) => {
    eprintln!("mistralrs validation error: {e}");
    break;
}
mistralrs::Response::ModelError(msg, _) => {
    eprintln!("mistralrs model error: {msg}");
    break;
}
```

**Why:** The actual `Response` enum has three distinct error variants:
- `InternalError(Box<dyn Error + Send + Sync>)` — engine-level failures
- `ValidationError(Box<dyn Error + Send + Sync>)` — malformed request
- `ModelError(String, ChatCompletionResponse)` — model returned an error with a
  partial response payload

None of them is named `Error`. Matching `Response::Error` would not compile.
All three are handled by logging to stderr and breaking the stream — consistent
with the "never panic" constraint from the original design.

---

### Bug 9 — `mistralrs::Model` is not `Clone`; mutex cloning broken

**Before:**
```rust
struct MistralRsBackendInner {
    model: Option<mistralrs::Model>,  // not Clone
}
// ...
let model = inner.model.clone()  // compile error
```

**After:**
```rust
struct MistralRsBackendInner {
    model: Option<Arc<mistralrs::Model>>,  // Arc is Clone
}
// ...
let model = self.inner.lock().unwrap().model.clone()?;
// yields Arc<mistralrs::Model> — cheap ref-count bump, not a deep copy
```

**Why:** `mistralrs::Model` is a heavyweight object (holds the loaded weights,
scheduler, tokenizer). It intentionally does not implement `Clone`. To release
the mutex before starting async work while still holding a reference to the
model, we wrap it in `Arc`. Cloning the `Arc` bumps the reference count (O(1),
no data copy) and the mutex can be released. The model lives until all `Arc`
clones are dropped. This is the standard Rust pattern for shared ownership of
non-`Clone` data across async tasks.

### Bonus Fix — `block_async` helper extracted

**Before:** Two copies of the
`tokio::runtime::Handle::try_current().ok().and_then(...).or_else(...)` pattern
appeared in `load_model` and `infer`, each ~10 lines. They also had subtle
differences (one used `.into()` incorrectly).

**After:**
```rust
fn block_async<F, T>(fut: F) -> Option<T>
where
    F: std::future::Future<Output = T> + Send + 'static,
    T: Send + 'static,
{
    if let Ok(handle) = tokio::runtime::Handle::try_current() {
        Some(tokio::task::block_in_place(|| handle.block_on(fut)))
    } else {
        tokio::runtime::Runtime::new().ok()?.block_on(async { Some(fut.await) })
    }
}
```

**Why:** The two-strategy async-bridging pattern (use existing runtime via
`block_in_place`, fall back to creating a temporary runtime) is correct but
tricky to get right. Extracting it into a named function makes both call sites
trivial and ensures the two strategies are identical. The `.into()` mis-usage
in the original `infer` path (which would have silently returned `None` in
some runtime configurations) is eliminated.

### Bonus Fix — `GgufModelBuilder` file list takes `Vec<impl ToString>`

**Before:**
```rust
mistralrs::GgufModelBuilder::new(path_str, vec![path_buf])
//                                              ^^^^^^^^ PathBuf: !ToString
```

**After:**
```rust
let file_str = path_buf.to_string_lossy().to_string();
mistralrs::GgufModelBuilder::new(path_str, vec![file_str])
```

**Why:** `GgufModelBuilder::new(model_id: impl ToString, files: Vec<impl ToString>)`
requires `ToString`. `PathBuf` implements `Display` but not `ToString` directly
in Rust's trait system (only types that implement `Display` get `ToString`
automatically, but `PathBuf`'s `Display` goes through `to_string_lossy` which
is lossy and not the direct impl path). The safe conversion is
`.to_string_lossy().to_string()` which replaces any non-UTF-8 bytes with `U+FFFD`.
On all target platforms (Linux, Windows, macOS), GGUF paths are UTF-8 in practice.
