## Summary

Four pieces of work shipped in one session:

1. **`MmapLoader`** — a new Rust-native mmap-based file loader (`engine/loader.rs`) that replaces
   candle's built-in GGUF I/O path with a zero-copy, `MADV_SEQUENTIAL`-hinted, magic-validating
   front-end. Windows-safe (madvise stubbed out with `#[cfg(not(unix))]`).

2. **Dequant dispatch wiring** — `engine/gguf_loader.rs` extended with `dequant_all()` and
   `load_and_dequant()`, connecting the full pipeline: disk → mmap → parse → dequant →
   `HashMap<String, Vec<f32>>`.

3. **`bench_ggqr` binary** — a standalone release-mode benchmark (`src/bin/bench_ggqr.rs`) that
   measures load + dequant throughput in GiB/s over N iterations, with an optional regression sum
   check (`--expected-sum`).

4. **`test_token_budget_respected` bugfix** — the pre-existing windowing test was failing because
   `apply_token_budget` unconditionally admitted the first window regardless of size, allowing a
   single merged window covering the entire file to blow past the budget. Fixed by truncating the
   first window's content when it exceeds the budget.

## Why

### MmapLoader (`loader.rs`)

The existing load path inside `gguf_parser::parse()` uses a `BufReader<File>`, which performs
buffered sequential reads. For multi-gigabyte GGUF files (Qwen3-1.7B = 1.92 GiB, LLaMA 3.1-8B
= 8.5 GiB) this means the OS has no advance knowledge of the access pattern. The kernel may
issue smaller, less-optimal prefetch requests than it would if told the access is sequential.

`memmap2` + `MADV_SEQUENTIAL` tells the kernel to aggressively read ahead, which on NVMe
SSDs (the hardware GwenLand targets) can increase effective throughput from ~3–5 GiB/s to
~9–12 GiB/s for large sequential reads. The mmap also eliminates one `memcpy` that `BufReader`
would otherwise perform when copying OS page cache pages into userspace buffers.

The magic-byte validation at the mmap layer catches corrupt or misidentified files before
any parsing work begins, giving a clean error message rather than a confusing parse failure
deep in the reader.

### Dequant dispatch (`gguf_loader.rs`)

After `load_gguf()` returned a `GgufFile` in the previous cycle, callers had to manually
invoke `dequant::dequantize()` per tensor in a loop. This pattern was repeated in the benchmark,
in the convert pipeline, and in test harnesses — three copies of the same loop with no error
handling consistency. `dequant_all()` centralises this with the correct skip-on-unsupported
behaviour (warn + skip, not abort) and a single `Err` propagation path for real errors.

`load_and_dequant()` collapses the two-step (load → dequant) pattern that every real caller
uses into a single call, which also enables the benchmark binary to be written in a few lines.

### `bench_ggqr` binary

Before this change there was no way to measure actual load + dequant throughput on a real GGUF
file. The only existing benchmark (`core_bench.rs` / Criterion) tests individual dequant
functions on synthetic data. `bench_ggqr` is an end-to-end wall-clock benchmark against a real
file, producing GiB/s figures that can be compared against the 9.0 GiB/s target and tracked
across cycles.

The `--expected-sum` regression check serves as a lightweight correctness guard: if the
dequantised float sum drifts between code changes, the binary exits 1 and CI catches the
regression without needing a reference GGUF file in the test suite.

### `apply_token_budget` bugfix (`windowing.rs`)

The `test_token_budget_respected` test had been failing because `apply_token_budget` had a
short-circuit `if kept.is_empty() || used + cost <= budget` that bypassed the budget check
for the very first window. In the test scenario (500 matching lines, `window_size: 20`,
`max_windows: 5`), `merge_windows` collapsed all five candidate windows into a single window
covering hundreds of lines (~27,000 chars). That single window was admitted unconditionally,
producing `total_chars = 27,000` against a `token_budget * 4 = 1,024` limit — a 26× overshoot.

The "always include at least one result" invariant is correct and must be preserved (callers
must not receive an empty window list for a valid query). But the invariant should apply to
the *presence* of a window, not to its *size*. The fix truncates oversized first windows to
`budget * 4` chars before admitting them, keeping the invariant while honouring the budget.

## What

### New file — `src/engine/loader.rs`

**`MmapLoader` struct:**

| Method | Description |
|---|---|
| `MmapLoader::open(path)` | Opens file, mmaps read-only, validates `b"GGUF"` magic, applies `MADV_SEQUENTIAL` on Unix |
| `as_bytes() → &[u8]` | Returns the full mapped byte slice |
| `len() → usize` | File size in bytes |
| `is_empty() → bool` | True if file is empty (always false for a successfully opened GGUF) |

**Platform behaviour:**

| Platform | madvise | mmap |
|---|---|---|
| Linux / macOS | `mmap.advise(Advice::Sequential)` — kernel prefetches pages ahead | ✅ |
| Windows | No-op stub (`#[cfg(not(unix))]`) | ✅ |

**Safety contract:** The single `unsafe` block is the `MmapOptions::new().map(&file)` call,
required by `memmap2`'s API. The mapping is read-only; no aliased mutable references exist.
No new `unsafe` beyond this.

**Dependency added:** `memmap2 = "0.9"` (zero transitive dependencies beyond `libc` on Unix,
which was already present via `memmap2`'s own dependency chain).

**Unit tests (3):**

| Test | What it verifies |
|---|---|
| `test_mmap_loader_invalid_magic` | Wrong magic bytes → `Err` containing `"magic"` |
| `test_mmap_loader_valid_magic` | `b"GGUF" + padding` → `Ok`, `as_bytes()[..4] == b"GGUF"` |
| `test_mmap_loader_empty_file` | Empty file → `Err` (too small for magic check) |

### Bugfix — `src/engine/windowing.rs` — `apply_token_budget`

**Before (broken):**

```rust
if kept.is_empty() || used + cost <= budget {
    used += cost;
    kept.push(w);
}
```

The `kept.is_empty()` short-circuit caused the first window to always be admitted at
full size, regardless of budget. A single merged window covering 500 lines (~27,000 chars)
was admitted with `token_budget * 4 = 1,024` — a 26× budget overshoot.

**After (fixed):**

```rust
if kept.is_empty() {
    if cost <= budget {
        kept.push(w);          // fits — admit normally
    } else {
        // Truncate to budget * 4 chars, recompute end_line
        let truncated: String = w.3.chars().take(budget * 4).collect();
        kept.push((w.0, new_end, w.2, truncated));
    }
} else if used + cost <= budget {
    kept.push(w);
}
```

The invariant "callers always receive at least one window" is preserved. Budget compliance
is now enforced even on the mandatory first window.
