# GwenLand — OOM-Safe Two-Mode mmap Loader (LoadMode::Eager / Lazy)

**Date:** 2026-06-08
**Scope:** `gwen-cli/packages/core/src/engine/loader.rs`, `gwen-cli/packages/core/src/engine/gguf_loader.rs`
**Type:** Feature — adaptive mmap load strategy + explicit mode override
**Status:** ✅ STABLE — 44 engine tests pass, zero errors, zero new warnings

---

## Summary

Extends `MmapLoader` with an OOM-safe two-mode loading strategy driven by a
new `LoadMode` enum. Before opening any file, the loader queries available
system RAM and compares it against the file size:

- **Eager** — RAM > file size: applies `MADV_SEQUENTIAL` + `MADV_WILLNEED`.
  The kernel prefetches the entire file into page cache before the first byte
  is accessed, maximising NVMe throughput on machines with headroom.
- **Lazy** — RAM ≤ file size: applies `MADV_SEQUENTIAL` only. Pages fault in
  on demand from SSD. Safe on 8 GB machines loading a 1.9 GB model when other
  processes are resident.

Both modes are no-ops on Windows (all madvise behind `#[cfg(unix)]`).
`load_gguf` and the new `load_gguf_with_mode` log the selected mode to stderr
so operators can confirm which path was taken. The regression doc comment on
`load_and_dequant` records the Qwen3-1.7B Q8_0 sum baseline for future
cross-check runs.

---

## Why

### The Problem

The previous loader always applied `MADV_SEQUENTIAL` unconditionally. This is
correct behaviour for machines where the model fits comfortably in available
RAM — but on a machine with 8 GB RAM running an IDE, browser, and inference
server simultaneously, available memory may be only 1–2 GB. Issuing
`MADV_WILLNEED` in that state asks the kernel to fault in ~2 GB of pages
immediately, which can cause the system to start swapping other resident pages
out, degrading the entire machine while the GGUF load is in progress.

The opposite failure: on a machine with 32 GB RAM loading a 1.9 GB model,
never issuing `MADV_WILLNEED` leaves ~1 GiB/s of potential prefetch throughput
on the table, because the kernel only faults in pages as they are touched
sequentially rather than saturating the NVMe read queue.

Neither `MADV_SEQUENTIAL` alone nor `MADV_WILLNEED` alone is universally
correct. The right hint depends on the relationship between available RAM and
file size at the moment of loading.

### Why sysinfo (not /proc/meminfo directly)

`sysinfo` is already a direct dependency (`sysinfo = "0.30"`) and its
`available_memory()` method abstracts over Linux (`/proc/meminfo`),
Windows (`GlobalMemoryStatusEx`), and macOS (`host_statistics64`). Using it
here is consistent with `memory_guard.rs`, `dry_run.rs`, and
`benchmark/memory.rs`, which all use the same `System::new()` +
`refresh_memory()` + `available_memory()` pattern. A direct `/proc/meminfo`
read would break on Windows; a custom platform abstraction would duplicate work
already done by sysinfo.

### Why expose `mode` on the struct

Callers — primarily `load_gguf`, `bench_ggqr`, and future diagnostics — need
to log which mode was selected to make throughput numbers interpretable.
A 9 GiB/s result means something very different in Eager vs Lazy mode. Making
`mode` a `pub` field on `MmapLoader` is the zero-overhead way to expose this:
no getter, no allocation, no trait bound.

### Why `open_with_mode` alongside `open`

Integration tests and the `bench_ggqr` binary need to force a specific mode
regardless of available RAM (e.g. always benchmark in Eager to measure
peak throughput, always benchmark in Lazy to measure worst-case). A separate
`open_with_mode` method preserves the ergonomic `open()` default while giving
callers full control without needing to mock the RAM detection path.

### Why Lazy is the safe fallback

If `sysinfo` returns 0 for available memory (a rare but documented failure mode
on some container environments), or if the file size query fails before
`detect()` is called, the code falls through to `Lazy`. Lazy is always safe —
it never asks the kernel to do more than it can. Eager on a constrained system
is the dangerous default; Lazy on a spacious system only costs throughput, not
stability.

## What

### New — `LoadMode` enum (`loader.rs`)

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum LoadMode {
    Eager,  // MADV_SEQUENTIAL + MADV_WILLNEED
    Lazy,   // MADV_SEQUENTIAL only
}
```

**`LoadMode::detect(file_size_bytes: u64) → LoadMode`**

| Condition | Result |
|---|---|
| `available_memory() > file_size_bytes` | `Eager` |
| `available_memory() ≤ file_size_bytes` | `Lazy` |
| `available_memory() == 0` (sysinfo error) | `Lazy` (safe fallback) |
| `file_size_bytes == u64::MAX` | `Lazy` (always) |

Uses `System::new()` + `sys.refresh_memory()` + `sys.available_memory()` —
identical pattern to the existing usage in `dry_run.rs`.

---

### Updated — `MmapLoader` struct (`loader.rs`)

**Struct change:**

```rust
// Before
pub struct MmapLoader {
    data: memmap2::Mmap,
}

// After
pub struct MmapLoader {
    data: memmap2::Mmap,
    pub mode: LoadMode,   // selected mode, exposed for logging/benchmarking
}
```

**New method:**

```rust
pub fn open_with_mode(path: &Path, mode: LoadMode) -> Result<Self, String>
```

Follows the same 5-step pipeline as `open()`:
1. Open `File`
2. `MmapOptions::new().map(&file)` (the only `unsafe` block — unchanged)
3. Validate `b"GGUF"` magic
4. Call `apply_madvise(&mmap, mode)`
5. Return `Self { data: mmap, mode }`

**Updated `open()`:**

Now stats the file first (`fs::metadata(path).map(|m| m.len())`) to get the
file size for `LoadMode::detect()`, then delegates to `open_with_mode`. The
stat uses `unwrap_or(0)` — if the stat fails, `detect(0)` returns Eager (0
bytes always fits in RAM), which is still correct because `open_with_mode` will
fail on the `File::open` call immediately after with a proper error message.

---

### Updated — `apply_madvise` helper (`loader.rs`)

```rust
// Before
fn apply_madvise_sequential(mmap: &memmap2::Mmap)          // unix
fn apply_madvise_sequential(_mmap: &memmap2::Mmap)         // not(unix)

// After
fn apply_madvise(mmap: &memmap2::Mmap, mode: LoadMode)     // unix
fn apply_madvise(_mmap: &memmap2::Mmap, _mode: LoadMode)   // not(unix)
```

Unix implementation:
```rust
let _ = mmap.advise(memmap2::Advice::Sequential);  // always
if mode == LoadMode::Eager {
    let _ = mmap.advise(memmap2::Advice::WillNeed); // Eager only
}
```

Return values ignored intentionally — madvise is a hint, not a guarantee.

### Extended — `gguf_loader.rs`

**`load_gguf` — mode logging added:**

```rust
pub fn load_gguf(path: &Path) -> Result<GgufFile, String> {
    let loader = MmapLoader::open(path)?;
    eprintln!("[gwenland] load mode: {:?}", loader.mode);
    gguf_parser::parse(path)
}
```

**New function — `load_gguf_with_mode`:**

```rust
pub fn load_gguf_with_mode(path: &Path, mode: LoadMode) -> Result<GgufFile, String> {
    let loader = MmapLoader::open_with_mode(path, mode)?;
    eprintln!("[gwenland] load mode: {:?} (explicit)", loader.mode);
    gguf_parser::parse(path)
}
```

## LoadMode::detect() on This Machine

`detect(100)` → **`Eager`**

The test machine has significantly more than 100 bytes of available RAM, so
any real GGUF file (minimum ~100 MB) will also resolve to Eager unless system
memory is severely constrained at load time. On an 8 GB machine with only
~500 MB free and a 1.9 GB model, `detect(2_000_000_000)` would return `Lazy`.
